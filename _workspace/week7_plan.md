##### 2-1. 왜 이 얘기를 지금 하는가 — 오늘 심을 씨앗 목록

- **예시/비유**: "우리가 쓴 코드는 글자(문자열)일 뿐인데, 그게 어떻게 네모·색·움직임으로 변하는가?" 한 줄 오프닝. 씨앗 목록은 불릿 5개 정도로 나열.
- **씨앗 연결**: 오늘 심을 씨앗 명시 — (1) 파이프라인 5단계 이름만, (2) reflow·paint 두 단어, (3) transform/opacity가 왜 좋은가, (4) 불변성과 리렌더 연결, (5) 실무 측정 도구 존재. 마지막 서브(2-7)에서 회수.
- **템플릿 일치**: week6 대면 2-1 "왜 이 얘기를 지금 하는가" 패턴 그대로.

##### 2-2. 파이프라인 5단계 개괄 — 두 단어만 머리에 넣기

- **왜 이 서브인가**: 파이프라인 전체 지도를 한 번 보여주되, **외우기를 강제하지 않음**. 씨앗 심기 톤 가이드의 "좋은 예"(reflow/paint 두 단어만)를 그대로 따름.
- **예시/비유**: 5단계 이름을 한 줄씩 나열만 — 파싱(Parsing) → CSSOM → 레이아웃(Layout) → 페인트(Paint) → 컴포지트(Composite). 각 단계는 "뭐 하는지" 한 줄 설명. 코드 예시 없음. "외울 필요는 없고 reflow·paint 두 단어만" 문구로 톤 잡기.
- **씨앗 연결**: 이 개괄이 2-3(reflow vs repaint), 2-4(transform/opacity), 2-5(불변성·리렌더), 2-6(측정 도구)의 전제가 된다.

##### 2-3. Reflow(Layout) vs Repaint — "위치·크기 다시 계산" vs "색 다시 칠하기"

- **왜 이 서브인가**: 오늘 대면의 두 단어(reflow, paint) 중 첫 번째를 깊게. 초보자가 "애니메이션이 버벅여요" 문제를 만날 때 가장 먼저 부딪히는 개념. JS에서 `element.style.width = '300px'`를 매 프레임 바꾸면 왜 느려지는지를 이름으로 설명할 수 있게.
- **예시/비유**: "책장 배치 바꾸기(reflow) vs 책 표지만 다시 칠하기(repaint)" 비유. 10줄 이내 코드 예시 하나 — `top/left`로 움직이는 경우와 `transform`으로 움직이는 경우의 결에 대한 맛보기(깊이는 2-4에서). "reflow가 비싸다"는 한 줄 결론.
- **씨앗 연결**: 2-4에서 "그래서 어떤 CSS 속성이 좋은가"로 즉시 이어짐.

##### 2-4. 왜 `transform`/`opacity`가 좋은가 — Composite만 유발

- **왜 이 서브인가**: 2-3의 이론을 실무 감각으로 연결. 초보자가 애니메이션 튜토리얼을 따라 할 때 "왜 `transform: translateX`를 쓰라고 하지?"를 해소. 면접에서도 자주 나오는 실전 감각.
- **예시/비유**: 5줄 이내 CSS 비교 예시.
  ```css
  /* Before — reflow + paint 매 프레임 */
  .box {
    left: 100px;
    transition: left 0.3s;
  }
  /* After — composite만 */
  .box {
    transform: translateX(100px);
    transition: transform 0.3s;
  }
  ```
  "두 코드는 눈에 보이는 결과는 똑같지만, 브라우저가 하는 일은 완전히 다르다" 한 줄.
- **씨앗 연결**: week10 React·week11 배포와 직접 연결되는 건 아니지만, **취업 시장 감각으로 중요**(CSS 최적화는 주니어 면접 단골). "이 감각은 면접에서 자주 묻습니다" 한 줄.

##### 2-5. 불변성(week7 비대면)과 렌더링 연결 — React 진입 씨앗 (가장 중요)

- **왜 이 서브인가**: **오늘 대면에서 가장 중요한 회수 포인트**. 비대면 week7(불변성)과 대면 week7(렌더링)을 하나의 감각으로 묶는 자리. 3주 뒤 week10 React 씨앗의 핵심.
- **예시/비유**: 비유 중심 + 짧은 의사 코드.
  - "React는 state가 바뀐 걸 어떻게 알까? 리스트를 한 글자씩 비교하면 너무 느리다. 그래서 React는 **참조(주소)가 바뀌었는지만** 본다. 참조가 그대로면 '안 바뀐 걸로' 간주하고 다시 그리지 않는다 — 리렌더 최적화."
  - "이게 비대면 week7의 불변성 얘기와 바로 연결된다. 기존 배열에 `push`로 요소를 추가하면 참조가 그대로라서 React가 변화를 놓친다. 그래서 **새 배열을 만들어 반환**하는 패턴을 쓴다."
  - 의사 코드 5줄 이내 (상세 설명 금지, 감각만):
    ```js
    // 나쁜 예 — 참조 그대로
    list.push(newItem);
    setList(list);
    // 좋은 예 — 새 배열
    setList([...list, newItem]);
    ```
- **씨앗 연결**: week10 React에서 "왜 `setState`에 새 객체/배열을 넘기는가" 배울 때 "아, 그때 그 얘기!"가 일어나는 핵심 지점. "지금 자세히 이해 못 해도 됩니다. `setList([...list, newItem])` 이 한 줄만 기억해두세요" 톤.

##### 2-6. 개발자 도구 Performance 탭 맛보기 — 실무에서 어떻게 측정하는가

- **예시/비유**: "F12 → Performance 탭 → Record 버튼 → 페이지에서 몇 초 상호작용 → Stop" 흐름만 말로 묘사. "타임라인에 긴 초록 막대(paint)·보라 막대(layout) 같은 게 나온다" 정도. "이게 뭔지 지금 알 필요 없고, '있다'만 기억하세요".
