##### 1-1. 핵심 리마인드 (3~5줄씩)

- **클로저 — 함수가 선언될 때의 환경을 기억한다**
  - 비유: "makeCounter를 호출할 때마다 `count`라는 메모지를 한 장씩 새로 발급한다. 반환된 내부 함수는 그 메모지를 손에 쥔 채 바깥 함수가 끝나도 계속 읽고 쓸 수 있다."
  - 한 줄 핵심: "외부 함수가 종료돼도 내부 함수가 외부 변수에 접근할 수 있는 것."
  - React 씨앗: "week10에서 `useState`가 돌아가는 기반이 이 클로저다. 지금은 이름만 넣어두세요."
  - 코드 3~4줄 스니펫: makeCounter 미니 버전.

- **Promise — 비동기 결과를 값(객체)으로 다룬다**
  - 비유: "택배 송장 같은 것. '지금은 배송 중(pending), 배송 완료되면 이 주소로 와서 받아가(then), 분실되면 이 번호로 전화(catch)'. 콜백으로는 송장을 계속 사람이 들고 다니며 전해야 했다면, Promise는 송장 그 자체가 도구."
  - Promise 3상태(`pending`/`fulfilled`/`rejected`)는 **이름만** 빠르게.
  - 한 줄 핵심: "콜백 지옥 = 중첩으로 인한 가독성 붕괴. Promise는 `.then()` 체이닝으로 수평화."

- **async/await — 비동기를 동기처럼 읽는 설탕**
  - 한 줄 핵심: "`async` 함수는 항상 Promise를 반환한다. `await`는 Promise 앞에서만 의미 있다."
  - 핵심 함정: **순차 await의 느림**. `const a = await A(); const b = await B();`는 A 끝난 뒤 B 시작. 의존성 없으면 `Promise.all([A(), B()])`로 병렬.
  - **오늘 대면 메인에 직결**: "fetch는 Promise를 반환한다. 그래서 `await fetch(...)`가 된다."

- **Q3 선별 이유**: 오늘 대면 메인 2-3(fetch)에서 **병렬/순차 선택**이 바로 API 호출 최적화의 첫 감각. "여러 API를 동시에 부를 수 있다면 `Promise.all`" 한 줄이 오늘 메인으로 자연 이행.

##### 2-1. 도입 — 웹 앱은 왜 서버와 통신해야 하는가 + 오늘 씨앗 목록

**왜 이 서브인가**: 비대면에서 "async/await fetch" 문법은 봤지만 "왜 웹 앱이 서버를 불러야 하는가"는 설명되지 않음. 오늘 대면의 첫 의문을 먼저 던지는 자리.

**예시/비유**:

- "여러분이 지금 보는 네이버 메인 페이지에서 뉴스 목록은 HTML에 박제된 게 아니다. 브라우저가 로드된 뒤 JS가 **별도 요청**을 서버에 보내고, 그 응답을 받아서 화면에 그린다. 이 '별도 요청'이 오늘의 주제."
- "즉 현대 웹앱은 '첫 HTML(뼈대) + JS가 찍어내는 수많은 API 요청(살)'의 구조. 이 구조 이름이 **SPA(Single Page Application)** — 지금은 이름만 들어두세요. week10에서 다시 만납니다."

- **CORS** — 한 단어만
- **토큰은 어디에 저장?** (2-6의 핵심 씨앗, week9 본편 회수)

##### 2-2. HTTP 기초 감각 — 요청과 응답, 메서드, 상태 코드

**왜 이 서브인가**: API 통신의 문법이 바로 HTTP. 메서드·상태 코드는 **면접 단골**이기도 하지만 면접 섹션이 폐지됐으므로 여기 인사이트로 흡수. "HTTP를 이름만 알면 `axios`·`fetch`·React Query 문서를 읽을 때 막히지 않는다" 감각.

**예시/비유**:

- **요청/응답 편지 비유**: "내가 서버에 편지(요청)를 보낸다. 서버는 답장(응답)을 돌려준다. 편지 봉투에 **주소(URL)**, **용건(메서드)**, **내용물(body)**, **동봉 서류(headers)**가 있고, 답장에도 **우체국 도장(상태 코드)** + **내용물(body)**이 찍힌다."
- **메서드 4개** (한 줄씩):
  - `GET` — "읽기. 주소만 있으면 된다. 주소창 치는 게 대부분 GET."
  - `POST` — "쓰기/만들기. 새 게시글 작성, 로그인 요청."
  - `PUT`/`PATCH` — "수정. PUT은 통째로, PATCH는 일부만 (구분 몰라도 오늘은 OK)."
  - `DELETE` — "삭제."
  - 실무 감각 한 줄: "GET으로 서버 상태를 바꾸면 안 된다. `/delete?id=1`을 GET으로 만들면 **week9 CSRF 본편에서 즉사 패턴**으로 다시 만난다. (씨앗)"
- **상태 코드 빈도 순**:
  - `200 OK` — 성공. 매일 본다.
  - `201 Created` — POST로 리소스 생성 성공.
  - `400 Bad Request` — 요청이 잘못됨(내가 보낸 JSON이 이상하다든지).
  - `401 Unauthorized` — **로그인 안 됨**. 토큰 없거나 만료.
  - `403 Forbidden` — 로그인은 됐는데 **권한 없음**.
  - `404 Not Found` — 주소 자체가 없음.
  - `500 Internal Server Error` — **서버 쪽 문제**. 내 잘못 아님.
  - 한 줄 감각: "4xx는 보낸 쪽(나) 잘못, 5xx는 받은 쪽(서버) 잘못. 둘을 구분하는 게 디버깅 첫걸음."

**씨앗 연결**:

- GET과 상태 변경 → week9 CSRF
- 401/403 → week9 인증·인가·토큰
- 404 → 프론트 라우팅 감각(week10)

##### 2-3. fetch 기본 패턴 — 비대면 async/await와 자연스럽게 연결

**왜 이 서브인가**: 비대면에서 봤던 `fetch`를 **실제로 한 번 펼친다**. "10줄 이내"로 제한해 씨앗 심기 톤 유지. 에러 처리 관행도 얇게.

**예시/비유**:

- **GET 요청 10줄 이내** (씨앗 심기 톤):
  ```js
  async function getUser(id) {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const user = await res.json();
    return user;
  }
  ```

  - "여기서 `await` 두 번 쓰는 게 자연스럽다고 느껴지면 오늘 대면은 절반 성공."
  - "`res.ok`는 `2xx`면 true. 이걸 **개발자가 직접 체크**해야 한다 — fetch는 **4xx/5xx에도 reject되지 않기** 때문(함정). 반면 axios는 자동 reject. 지금은 이름만 들어두세요 — week10 React 쓸 때 axios·TanStack Query 나옵니다."
- **POST 요청 짧은 예**:
  ```js
  await fetch("/api/posts", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ title: "hi" }),
  });
  ```

  - "`Content-Type` 헤더를 안 붙이면 서버가 JSON인 줄 모른다. 실무 함정 1번."

**씨앗 연결**:

- `await fetch` 패턴 → 비대면 week8 3섹션과 정확히 같은 모양. 비대면→대면 다리.
- axios / TanStack Query 이름 → week10 대면(React 생태계)
- fetch 함정(4xx reject 안 됨) → week10에서 "왜 TanStack Query가 쉬운가"의 복선

##### 2-4. CORS가 뭘 막나 — "다른 도메인 API 호출은 기본적으로 차단"

**왜 이 서브인가**: **CORS는 입문자 3대 좌절 요소**(설정 에러 메시지로 좌절). 이름만 심어두면 나중에 마주칠 때 "아, 그 얘기!"가 된다. 심화는 안 함.

**예시/비유**:

- **동일 출처 정책(Same-Origin Policy) 비유**: "브라우저는 기본적으로 **내 집(도메인) 식구가 아니면 편지를 못 뜯게** 한다. 악의적인 사이트가 내 뱅킹 계정에 접근하는 걸 막기 위한 안전장치."
- **언제 CORS 에러를 보나**:
  - 프론트 `localhost:3000`이 백엔드 `localhost:8000`에 요청 → **포트가 다르면 다른 출처로 간주**. CORS 에러.
  - 에러 메시지 샘플 (실제 콘솔에 찍히는 모양 그대로):
    ```
    Access to fetch at 'http://localhost:8000/api/users'
    from origin 'http://localhost:3000' has been blocked
    by CORS policy
    ```
  - "이 메시지를 언젠가 반드시 본다. 그때 '아 이거!' 하면 성공."
- **해결 감각** (로우 터치, 한 줄씩):
  - 정석: **서버가 `Access-Control-Allow-Origin` 헤더로 허용**. 프론트가 할 일 아님.
  - 개발 환경: Next.js/Vite **dev proxy**로 우회 (이름만). week10에서 다시.
- **실무 감각 한 줄**: "CORS는 **서버의 책임**이지만, 프론트 개발자가 `백엔드 왜 안 열어줘`를 설명할 수 있어야 한다. 팀 협업 기본기."

**씨앗 연결**:

- CORS ≠ CSRF 방어 (week9 보안 본편에서 명시적으로 구분 다룸)
- dev proxy → week10 React/Vite/Next 생태계
- **중요**: "CORS는 브라우저가 막는다. Postman/curl에서는 에러 안 남" — week9에서 "프론트 검증은 무의미"로 이어지는 복선.

##### 2-5. REST 감 — 자원과 메서드의 조합

**왜 이 서브인가**: REST는 정확한 정의가 복잡하지만 실무에서 읽고 쓰는 API 주소의 **대부분이 REST 스타일**. **직관만** 심는다.

**예시/비유**:

- **URL은 "자원"을 가리킨다** 감각:
  - `GET /users` — 유저 목록 읽기
  - `GET /users/123` — 123번 유저 읽기
  - `POST /users` — 유저 새로 만들기
  - `PUT /users/123` — 123번 유저 통째 수정
  - `DELETE /users/123` — 123번 유저 삭제
  - "동사를 URL에 넣지 않는다 (~~`/getUser?id=123`~~ ❌). 동사는 **메서드**가 맡는다."
- **한 단계 더** (씨앗): "`GET /users/123/posts`처럼 **자원이 자원을 소유**하는 중첩 구조도 자주 본다. 이건 week10 React 배울 때 API 설계 감각으로 다시."
- **실무 감각**: "완벽한 RESTful은 드물다. 대부분 '대충 REST 스타일'. 팀마다 관행이 다르니 **팀 컨벤션이 있으면 그걸 따른다** (week6 대면 팀 컨벤션 회수)."
- **GraphQL 이름만 언급** (1줄): "React 커뮤니티에서 GraphQL도 쓴다. 하나의 엔드포인트에 **쿼리를 보낸다**. 지금은 이름만."

**씨앗 연결**:

- 팀 컨벤션 → week6 회수
- GraphQL → week10 React 생태계
- CSRF(CVE-2025-32354) → week9 보안 본편에서 "Zimbra GraphQL CSRF" 사례로 회수

##### 2-6. [핵심 씨앗] 토큰은 어디에 저장? — week9 보안 본편의 심장

**왜 이 서브인가**: **오늘 대면의 심장**. week9 보안 본편(localStorage vs HttpOnly 쿠키, "XSS 한 방 = 토큰 탈취")에서 직접 회수되는 가장 중요한 씨앗. 한 줄 공포로 심는다.

**예시/비유**:

- **로그인 흐름 한 번 정리** (3줄):
  - 유저가 ID/PW로 로그인 요청 → 서버가 검증 → **토큰(JWT 같은 문자열) 발급** → 이후 요청마다 이 토큰을 동봉해서 "나 맞아요" 증명.
- **토큰을 어디에 두는가** (표 or 비교):
  - **localStorage** — JS로 쉽게 읽고 쓸 수 있어서 **튜토리얼의 90%가 여기 쓴다**. 하지만…
  - **HttpOnly 쿠키** — JS에서 **못 읽게** 설정된 쿠키. 브라우저가 자동으로 요청에 동봉.
- **한 줄 공포** (인용구 강조):
  > localStorage에 토큰 = XSS 한 방 = **Game Over**.
  - "XSS는 남의 JS가 내 페이지에서 실행되는 공격. localStorage는 **모든 JS가 읽을 수 있으므로** 토큰 전체가 그 JS에 넘어간다. HttpOnly 쿠키는 JS가 못 읽으니 한 단계 더 버틴다."
- **모범답 감각** (한 줄):
  - "access token은 React 메모리(state), refresh token은 HttpOnly·Secure·SameSite 쿠키 — 이 패턴이 현업 표준."
- **지금 외울 필요 없음 강조**: "이름만. '아 토큰 저장 위치로 공격이 달라지는구나'만 머리에 남으면 충분. **다음 주 week9에서 이 한 줄이 본편의 심장**이 됩니다."
- **AI 코딩 경고 한 줄**: "AI가 만드는 로그인 코드 90%가 `localStorage.setItem('token', ...)`이다. 이유는 입문 튜토리얼 문법이 AI 학습 데이터의 다수이기 때문. week9 대면에서 실제 사례와 숫자로 다시."

**씨앗 연결** (가장 중요):

- **week9 대면 "바이브코더가 모르는 웹 보안" 본편** 전면 회수
- XSS 한 방 감각 → week9 XSS·DOMPurify·`dangerouslySetInnerHTML` 섹션
- HttpOnly·Secure·SameSite → week9 세션 쿠키 3종 세트
- Lovable 170개 앱 토큰 유출 사례 → week9 오프닝 훅 숫자
