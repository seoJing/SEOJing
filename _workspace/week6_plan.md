### 본문 섹션 개요 (현행 유지 + 최소 다듬기)

> 현행 구조(섹션 1 객체 / 섹션 2 함수 / 섹션 3 스코프)가 얄코 "제어문·스코프·함수·객체 기본" 범위를 자연스럽게 재배열한 형태로 **이미 적절**. 사용자 제약 #5에 따라 구조 변경은 하지 않음.

1. **섹션 1 — 객체: JS에서 데이터를 담는 기본 그릇** (현행 유지)
   - 의도: 다른 언어(클래스 선행 정의) 경험자에게 "JS는 클래스 없이 객체 리터럴로 바로 만든다"는 차이 전달.
   - 서브 (현행 그대로):
     - 1-1. 객체 리터럴 생성, 프로퍼티 추가/삭제 데모
     - 1-2. 프로퍼티 접근: 점 표기법 vs 대괄호 표기법 (공백 키, 변수 키)
     - 1-3. 원시값 vs 객체: 값 복사 vs 참조 복사 (+ spread 얕은 복사 예고)
   - **변경사항**: 없음. 섹션 1 마지막 단락 끝에 "객체 복사의 함정(얕은 복사 vs 깊은 복사)은 week7에서 `Object.freeze`와 함께 다시 만납니다" 한 줄 추가로 week7 연결 강화 — **선택 사항**.

2. **섹션 2 — 함수: 일급 객체로서의 함수** (현행 유지)
   - 의도: 다른 언어(함수가 클래스 메서드 종속) 경험자에게 "JS 함수는 값처럼 다룰 수 있다"는 일급 객체 개념 전달. 이 감각이 week7 프로토타입·week10 고차함수·week11 클로저의 전제.
   - 서브 (현행 그대로):
     - 2-1. 일급 객체 3대 특성 데모 (변수 할당 / 인자 전달 / 반환값)
     - 2-2. 함수 선언식 vs 함수 표현식 — 호이스팅 차이, `const` + 표현식 실무 권장
     - 2-3. 화살표 함수 — 간결 문법, **this 바인딩 차이는 "다음 주에서" 예고만** (사용자 제약 #5에 따라 현행 유지)
     - 2-4. 콜백 함수 — `map`/`filter`/`reduce` 짧은 예시
   - **변경사항**: 없음. 2-3 화살표 함수의 "다음 주 this" 예고는 **실제로는 week9 클래스/this 주차에서 처음 등장**하므로 "몇 주 뒤 this 관련 주제에서 다시 만납니다"로 표현을 완화할지 post-writer 판단(⚠️ 현재 본문에 "다음 주 this에서 자세히 다룹니다"라고 되어 있는데, week7 실제 내용은 "불변성·프로토타입·타입 체크"라 `this` 심화는 아님 → **표현 수정 권장**).

3. **섹션 3 — 스코프: 변수가 살아있는 범위** (현행 유지)
   - 의도: 다른 언어 경험자가 가장 당황하는 지점(함수 스코프/렉시컬 스코프) 정리. 렉시컬 스코프는 week11 클로저의 전제.
   - 서브 (현행 그대로):
     - 3-1. var의 함수 스코프 — if 블록 무시 예시
     - 3-2. 스코프 체인 — 안→밖 탐색 코드
     - 3-3. 렉시컬 스코프 — 호출 위치 vs 선언 위치, 클로저 예고
   - **변경사항**: 없음. 섹션 3-3 마지막 단락의 "다음 주에 배울 클로저(Closure)의 핵심 원리입니다" 문구는 **실제로는 week11**에서 클로저를 다루므로 "몇 주 뒤 클로저 주차에서 다시 만납니다"로 수정 권장. (⚠️ post-writer 주의사항에 기재)

### 섹션 구성 (3섹션 고정 템플릿 기반)

#### 섹션 1. 이번 주 사전학습 점검 (약 15~20분)

- 대상: **같은 번호 비대면 week6** = 객체 · 함수(일급 객체) · 스코프(함수/블록/렉시컬)
- 구성:
  - 1-1. **핵심 리마인드** (구술 설명 옆에 두고 읽는 용도, 비유·코드 스니펫 중심):
    - **참조 복사의 함정** (기존 대면 유지): `const a = {x:1}; const b = a; b.x = 99;` → `a.x === 99`. 원시값 복사와 대비해 한 줄 설명.
    - **일급 객체가 왜 중요한가 — React와 연결** (기존 대면 유지 + 씨앗 강화): `<button onClick={handleClick}>` 한 줄 JSX 스니펫만 노출, "React에서 이벤트 핸들러 전달되는 게 함수가 일급 객체라서 가능하다"는 한 문장. **10줄 이하 JSX 1개만** — 씨앗 심기 톤 준수.
    - **함수 선언식 vs 표현식 호이스팅** (추가 권장 — 현재 대면 섹션 1에는 없고 섹션 2 과제 풀이 Q1에만 있음): 6줄 이내 스니펫(`foo()` 선언식 OK, `bar()` 표현식 TypeError). "호이스팅은 됐지만 값이 undefined라 TypeError" 한 줄 요약.
    - **렉시컬 스코프 — 전학생 비유** (기존 대면 유지): `foo`는 전역에서 작성됐으므로 `bar` 내부에서 호출해도 전역의 `x`를 참조. "학생이 다른 반에 놀러 가도 소속 반 규칙을 따른다" 비유 유지.
  - 1-2. **사전학습 인증 미션 풀이** — week6 비대면 인증 미션 Q1~Q4 중 **Q1(호이스팅 TypeError)과 Q2(렉시컬 스코프) 두 문제만 선별 풀이**. Q3(참조 복사)와 Q4(IIFE)는 섹션 1-1 "핵심 리마인드"에서 이미 다뤄지거나 스킵 진단에서 학습자가 풀어본 문제이므로 "비대면 자료 참조 + 댓글 제출"로 넘김.
    - Q1 풀이: 정답 전개 "함수 선언식은 전체 호이스팅, `var` 함수 표현식은 선언만 호이스팅되어 `undefined` 상태에서 호출하면 TypeError. `const` 표현식은 TDZ로 먼저 ReferenceError." + 흔한 오답("호이스팅이 안 돼서 에러"가 아니라 "호이스팅은 됐는데 값이 undefined여서 에러"가 핵심)
    - Q2 풀이: 정답 `"global"` + 렉시컬 스코프(정적 스코프) 정의 + "bar 내부에서 호출했으니 bar의 x가 나올 것"이라는 동적 스코프식 사고가 흔한 오답.
  - 1-3. **막힌 사람 클리닉 (상시)** — 비대면에서 막힌 포인트 자유 공유. "이해 안 됐어요"가 스터디의 기본값. (week5 대면과 동일 문구 유지)
- 톤: 구술 설명용. 리스트보다 비유/코드/예시 중심. 기존 대면 week6 섹션 1 "저번 주 복습" + 섹션 2 "저번 주 과제 정답 및 풀이" 내용을 **이 섹션 1 하나로 압축**.

#### 섹션 2. [대면 메인] 팀 프로젝트 컨벤션·그라운드룰 + Git/PR 기초

> 목표: "코드를 혼자 쓸 때와 팀으로 쓸 때 뭐가 다른가"를 단어·감각·실제 예시 수준에서 심어둔다. Git 명령어 나열은 금지 — "왜 필요한가"만. 씨앗 심기 톤 준수(새 용어 한 번에 3개 이하, 코드 10줄 이내, "왜" 먼저, 연결 고리 명시).

##### 2-1. 왜 이 얘기를 지금 하는가 (도입, 3~5분)

- "비대면에서 함수·변수 이름 짓는 걸 막 배웠죠. 오늘은 그 이름이 **혼자 쓸 때와 팀으로 쓸 때 어떻게 의미가 달라지는지**를 봅니다."
- 오늘 심을 씨앗 목록을 먼저 제시 (5~6개):
  - 팀으로 코드 쓰기는 왜 어려운가 (컨벤션이 필요한 이유)
  - 네이밍 컨벤션 — 함수·변수 이름이 팀 속도를 정한다
  - Git 기초 감각 — commit·branch·PR이 왜 필요한가 (명령어 아님)
  - PR 리뷰 문화 — 리뷰는 트집이 아니라 품질 합의
  - 그라운드룰 예시 — 실제 팀이 어떻게 운영되는지
  - AI와 팀 — AI 코드를 PR에 올릴 때의 매너
- 톤: "오늘 전부 이해 못 해도 됩니다. 이 단어들이 몇 주 뒤 팀플에서 다시 올라옵니다." 한 줄.
- **왜 이 시점인가** 한 문단 추가: 스터디 후반부(week10~11)에 팀 프로젝트가 본격화되기 전에, **지금 감각을 심어두지 않으면 "Git이 어려워서 기능 구현에 들어가지 못하는" 상황이 발생**한다. 지금 심는 씨앗이 팀플 시점에 싹트는 것이 목표.

##### 2-2. 팀으로 코드 쓰기는 혼자 쓰기와 뭐가 다른가

- 구성 (한 문단 + 짧은 비교):
  - 혼자 코드: 내가 쓴 이름·스타일·순서를 내가 이해하면 됨. 내일의 나조차 못 알아보는 수준으로 안 쓰면 그럭저럭 돌아감.
  - 팀 코드: **"내일의 다른 사람"이 읽는 코드**. 내 머릿속 맥락이 없는 사람도 바로 이해할 수 있어야 함. 안 그러면 팀 속도가 느려짐.
  - 비유: "혼자 쓰는 일기장"과 "여러 명이 같이 쓰는 위키"의 차이. 위키는 제목·카테고리·작성 규칙이 필요.
  - 실전 감각: 주니어가 입사해서 가장 먼저 배우는 게 "이 팀의 코드 스타일"이다. 문법보다 스타일이 먼저 부딪히는 벽.
- **씨앗**: week9 보안에서도 "다른 사람이 내 코드를 쓴다"는 관점이 중요해진다(외부 입력을 믿지 않기 등). 지금 감각이 그때 확장된다.
- 코드 예시: **없음** — 추상 개념이지만 2-3부터 구체 사례가 쏟아지므로 여기서 코드 없이 감만 잡기.

##### 2-3. 네이밍 컨벤션 — 함수·변수 이름이 팀 속도를 정한다

- 도입: 비대면 week6에서 배운 함수·변수·프로퍼티 이름 짓기가 **혼자 쓸 때는 "에러만 안 나면 OK"지만, 팀에서는 "읽는 속도가 개발 속도"**.
- 구성 (짧은 규칙 + 비교 스니펫 중심, 규칙은 외우는 게 아니라 "이런 감각이 있다"만):
  - 규칙 ① **불리언은 `is`/`has`/`can`으로 시작**: `loading` 대신 `isLoading`, `admin` 대신 `isAdmin`. React에서 매일 보게 될 패턴.
  - 규칙 ② **함수는 동사로 시작**: `user()` 대신 `getUser()`, `submit` 대신 `handleSubmit`. React에서 이벤트 핸들러는 `handleXxx` 관례.
  - 규칙 ③ **이름으로 단위를 표시**: `timeout` 대신 `timeoutMs`, `width` 대신 `widthPx`. 단위 불일치 버그 예방.
  - 규칙 ④ **약어 남용 금지**: `usrLstCnt` 대신 `userListCount`. 타이핑 시간보다 읽는 시간이 훨씬 길다.
- 예시 스니펫 1개 (10줄 이내, 비포/애프터):

  ```js
  // Before — 팀원이 10초 동안 멈춰야 함
  function check(u, t) {
    if (u.a > t) return true;
    return false;
  }

  // After — 읽자마자 이해됨
  function isUserOlderThan(user, ageThreshold) {
    return user.age > ageThreshold;
  }
  ```

- **씨앗**: week10에서 React 컴포넌트·훅 이름 짓기(`useXxx`, 컴포넌트는 PascalCase)로 다시 등장. 지금 감각이 그때 기반.
- 연결 고리: "네이밍은 AI 시대에 **더 중요**해졌어요. AI가 써준 코드를 내 팀 스타일로 다시 이름 붙이는 게 주니어의 일 중 하나."

##### 2-4. Git 기초 감각 — commit·branch·PR이 왜 필요한가

- **금지 사항 먼저 공표**: "오늘은 Git 명령어를 외우지 않습니다." `git add`·`git commit`·`git push` 같은 명령어는 검색 한 번이면 나옴. **감각**만 심는다.
- 구성 (3단 설명 — 각 1~2문단):
  - **commit = "세이브 포인트"**: 게임에서 세이브 안 하고 보스 만나면 망하죠. 작업 중간중간 "여기까지는 확실히 동작함"을 찍어두는 것. 문제가 생기면 그 지점으로 돌아갈 수 있음.
  - **branch = "평행우주"**: main은 원본, 내 작업은 별도 우주에서 진행. 내가 실수해도 main은 안 망가짐. 다 만들고 나서 "이 평행우주를 main에 합치자"고 제안하는 게 PR.
  - **PR(Pull Request) = "합쳐달라는 제안서"**: 평행우주에서 완성한 작업을 main에 합치기 전에, 팀원들이 **리뷰하고 합의**하는 자리. GitHub·GitLab 같은 도구가 이걸 UI로 해줌.
- 왜 필요한가 (한 문단): "혼자 만들면 그냥 파일에 직접 써도 돼요. 근데 팀 3명이 같은 파일을 동시에 만지면 충돌이 나고, 누가 뭘 왜 바꿨는지 흔적이 사라져요. Git은 **'누가 언제 왜 뭘 바꿨는지'를 전부 기록**하고, 충돌을 드러내서 팀이 합의할 수 있게 해주는 도구."
- 코드 예시: **없음**. 명령어는 일부러 안 보여줌. 대신 GitHub PR 화면 스크린샷을 구두로 묘사 정도.
- **씨앗**: week11 배포에서 **Vercel이 GitHub와 연동**되어 "main에 머지되면 자동 배포"가 된다는 얘기가 나옴. 오늘의 PR/merge 감각이 그때 필요.
- 연결 고리: "실제 명령어 실습은 동아리 팀 프로젝트 시점에 해요. 오늘은 'commit·branch·PR이 뭐 하는 거구나'만."

##### 2-5. PR 리뷰 문화 — 리뷰는 트집 잡기가 아니라 품질 합의

- 도입: "PR이라는 게 기술적 합치기만은 아니에요. **'이 코드 괜찮은지 팀원들이 같이 보는 자리'**. 이게 왜 중요한지 얘기할게요."
- 구성 (3단 관점):
  - 관점 ① **리뷰는 공격이 아니다**: "이 코드 왜 이렇게 짰어요?"는 트집이 아니라 질문. 작성자가 "왜"를 설명하면서 자기 코드를 다시 보게 됨. **리뷰어도 배우고, 작성자도 배움.**
  - 관점 ② **리뷰에서 걸러지는 것들**:
    - 네이밍 혼란 (2-3과 연결)
    - 중복 코드
    - **"이 코드는 안전한가"** — 사용자 입력을 검증하지 않거나, 비밀번호를 콘솔에 찍는 것 같은 실수 (week9 보안 연결)
    - 테스트 누락
  - 관점 ③ **좋은 PR 설명 = 좋은 커밋 메시지 + 스크린샷**: "이 PR이 뭘 왜 고쳤는지, 리뷰어가 어디를 봐야 하는지"를 작성자가 먼저 설명. 설명 없이 "PR 확인 부탁드려요" 하나만 달랑 올리면 리뷰 품질이 떨어짐.
- 실전 예시 한 줄: "좋은 리뷰는 `이 함수, getUser 대신 fetchUserById가 더 명확할 것 같아요` 같은 구체 제안. 나쁜 리뷰는 `뭔가 이상한데요` 같은 막연한 불만."
- 코드 예시: **없음**. 대신 **좋은 PR 설명 템플릿**을 짧게 제시(5~7줄):

  ```
  ## 무엇을
  - 로그인 실패 시 에러 메시지를 콘솔이 아닌 UI에 표시

  ## 왜
  - 사용자가 로그인 실패 사유를 알 수 없다는 제보 (#42)

  ## 리뷰 포인트
  - 에러 메시지 문구가 적절한가
  ```

- **씨앗**: week9 보안 대면에서 "AI가 기본적으로 막아주지 않는 영역"을 다룰 때, **"PR 리뷰에서 누군가는 이 코드가 안전한지 물어야 한다"**는 관점이 이어진다.
- 연결 고리: "면접에서 '협업 경험' 물어보면 'PR 리뷰를 이렇게 받았고, 이런 피드백으로 코드가 이렇게 바뀌었다'라고 답할 수 있는 게 가장 구체적이에요."

##### 2-6. 그라운드룰 예시 — 실제 팀이 어떻게 운영되는가

- 도입: "지금까지 개념만 얘기했으니, **실제 팀에서 쓰는 그라운드룰 샘플** 1~2개만 보여드려요."
- 구성 (짧은 샘플 두 세트):
  - **샘플 ① 커밋 메시지 규칙** — Conventional Commits 간단 버전:
    - `feat: ~` (새 기능), `fix: ~` (버그 수정), `docs: ~` (문서), `refactor: ~` (리팩터), `chore: ~` (기타)
    - "뭘 왜 바꿨는지가 커밋 제목만 봐도 보여야 한다"
    - 예시 3줄:
      ```
      feat: 로그인 실패 에러 메시지 UI 표시
      fix: 버튼 클릭 시 2번 submit되는 문제 해결
      docs: README에 로컬 개발 세팅 추가
      ```
  - **샘플 ② 브랜치 네이밍 규칙**:
    - `feature/login-error-ui`, `fix/double-submit`, `refactor/user-api`
    - "타입/주제" 두 부분으로 하이픈 구분. 한국어 금지(툴 호환성).
- 톤: "이 규칙이 정답은 아니에요. **팀마다 다르지만 핵심은 '팀원이 합의한 규칙이 있다'는 것**. 그라운드룰은 '이 팀이 어떻게 일하는지'를 정하는 문서."
- **씨앗**: week10 React 프로젝트 씨앗 심을 때 "Next.js 기본 디렉토리 구조"를 언급하는데, 그것도 일종의 컨벤션 — 프레임워크가 규칙을 강제하는 쪽. 오늘의 팀 규칙과 다른 결.
- 연결 고리: "실제 팀에 들어가면 첫 PR에서 이 규칙 안 지켜서 리뷰 멘트 받는 경우가 많아요. 미리 봐두세요."

##### 2-7. AI와 팀 — AI가 쓴 코드를 PR에 올릴 때 (week5 AI 씨앗 회수)

- 도입: "week5에서 'AI 시대에 왜 JS를 배우나'에서 꺼낸 얘기 기억나세요? 오늘은 **AI가 쓴 코드를 팀에 올릴 때 뭘 조심해야 하는지** 짧게 다뤄요."
- 구성 (3개 포인트):
  - 포인트 ① **AI가 쓴 코드도 내 책임**: PR을 올리는 순간 그건 **내 코드**. "AI가 써준 거라서 잘 모르겠어요"는 리뷰에서 통하지 않음. 이해한 코드만 올린다.
  - 포인트 ② **AI 표기는 팀마다 합의**: 일부 팀은 커밋 메시지에 `Co-Authored-By: AI` 표기를 요구, 일부는 "코드 품질만 맞으면 상관없다"는 입장. **"AI 썼어요?"라고 물어보는 게 매너**이고, 팀 그라운드룰에 이걸 넣는 경우도 늘고 있음.
  - 포인트 ③ **AI가 기본적으로 안 챙기는 것**: week5에서 본 친구 게임 사례 — AI는 "동작하는 코드"에 집중하고, **보안·엣지 케이스·팀 컨벤션**은 기본적으로 챙기지 않음. 리뷰어가 이걸 본다는 것이 핵심 역할.
- 논쟁 포인트 한 줄: "'AI 표기가 필요한가'는 실제로 의견이 갈리는 주제예요. 정답은 없고, **팀이 합의한 방식이 정답**." — 대면에서 짧게 의견 교환 가능.
- **씨앗**: week9 보안 대면에서 "AI가 안 챙기는 영역"을 본격적으로 푼다. 오늘은 "PR 올릴 때 그 관점이 필요하다"는 감각만.
- 코드 예시: **없음** — 논의 주제라 말로.

##### 2-8. 앞으로 스터디에서 어떻게 연결되는가 (씨앗 심기 마무리)

- 오늘 심은 씨앗 재정리 (학습자가 가져갈 키워드만):
