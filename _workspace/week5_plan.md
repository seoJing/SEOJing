       - `if (true) { var x = 10; } console.log(x);` 예시 그대로.
     - 2-2. **var의 문제점 2: 호이스팅** (기존 유지 + 시각화 강화)
       - 기존 "JS 엔진이 실제로 실행하는 방식" 주석 코드 유지.
       - 추가: **실행 컨텍스트 도식 설명 단락** — "생성 단계 → 실행 단계" 2단계로 JS 엔진이 스코프를 훑는다는 개념을 말로 설명. 코드는 없고 서술만, 5~8줄 Paragraph 정도.
       - 추가: "왜 `var`만 `undefined`가 되는가" — 선언 호이스팅 시 `var`는 초기화까지 함께 이뤄지고, `let`/`const`는 선언만 이뤄진다는 차이를 명시.
     - 2-3. **TDZ(Temporal Dead Zone) 시각화** (신규 서브, 섹션 3 삭제로 생긴 자리 활용)
       - 의도: "호이스팅은 되지만 초기화 전"이라는 개념을 코드 3~4개 스텝으로 풀어 보여줌.
       - 구성:
         - 코드 A: `console.log(a); let a = 1;` → ReferenceError
         - 코드 B: 함수 안에서 `let`이 TDZ에 놓이는 시점을 주석으로 단계 표시
         ```js
         function example() {
           // ── 여기부터 TDZ 시작 (a가 선언은 됐지만 초기화 전)
           console.log(a); // ReferenceError
           let a = 1;
           // ── 여기부터 TDZ 끝 (a 초기화 완료)
           console.log(a); // 1
         }
         ```

         - 코드 C: `typeof` 연산자도 TDZ에서는 에러 — `typeof a; let a;` 주의사항 한 줄.
       - 마무리 단락: "TDZ 덕분에 `let`/`const`는 호이스팅 덕에 일어나던 `undefined` 출력 버그가 근본적으로 차단됩니다."
     - 2-4. **var의 문제점 3: 중복 선언 허용** (기존 유지, 순서만 뒤로)
     - 2-5. **결론: 언제 뭘 쓸까** (기존 유지, 표도 유지)
       - 표 하단에 "실무 기본값: const → 재할당 필요 시 let. var는 레거시 코드 읽을 때만" 한 줄 유지.

#### 교체 문제 1 (기존 #4 falsy 자리)

- **질문**: JavaScript에서 변수에 값을 재할당할 수 없는 키워드는 무엇일까요?
- **선택지**:
  1. `var`
  2. `let`
  3. `const`
  4. 셋 다 재할당 가능
- **정답**: 2 (인덱스 2 = `const`)
- **해설**: `const`는 한 번 할당한 값을 다시 바꿀 수 없습니다. `var`와 `let`은 재할당이 가능합니다. 단, `const`로 선언한 객체의 내부 프로퍼티(`obj.name = 'hi'`)는 변경할 수 있습니다. 재할당과 내부 값 변경은 다른 개념입니다.

#### 교체 문제 2 (기존 #5 `'5'-1` 자리)

- **질문**: 다음 중 JavaScript의 **원시 타입(Primitive Type)**이 **아닌** 것은 무엇일까요?
- **선택지**:
  1. `string`
  2. `boolean`
  3. `object`
  4. `undefined`
- **정답**: 2 (인덱스 2 = `object`)
- **해설**: JS의 원시 타입은 `number`, `string`, `boolean`, `undefined`, `null`, `symbol`, `bigint` 7가지입니다. `object`는 참조 타입이며 배열·함수·일반 객체가 모두 `object`에 해당합니다. 원시 타입은 값 자체가 복사되고, 참조 타입은 메모리 주소가 복사됩니다.

#### 교체 문제 3 (기존 #6 `==`/`===` 자리)

- **질문**: JavaScript에서 `typeof` 연산자의 결과로 **얻을 수 없는** 문자열은 무엇일까요?
- **선택지**:
  1. `'number'`
  2. `'string'`
  3. `'array'`
  4. `'function'`
- **정답**: 2 (인덱스 2 = `'array'`)
- **해설**: 배열은 객체의 한 종류이므로 `typeof [] === 'object'`입니다. `typeof`가 반환하는 값은 `'number'`, `'string'`, `'boolean'`, `'undefined'`, `'object'`, `'function'`, `'symbol'`, `'bigint'` 총 8가지이며 `'array'`나 `'null'`은 없습니다. 배열 확인은 `Array.isArray()`를 사용해야 합니다.

#### 교체 문제 4 (기존 #10 단축 평가 자리)

- **질문**: `let`과 `const`로 같은 이름의 변수를 다시 선언하면 어떻게 될까요?
- **선택지**:
  1. 경고 없이 덮어써진다
  2. `undefined`가 할당된다
  3. `SyntaxError`가 발생한다
  4. 이전 값이 유지된다
- **정답**: 2 (인덱스 2 = `SyntaxError`가 발생한다)
- **해설**: `let`과 `const`는 같은 스코프에서 중복 선언이 금지되어 있어 `SyntaxError: Identifier 'x' has already been declared`가 발생합니다. 반면 `var`는 중복 선언이 허용되어 조용히 덮어써집니다. 이 차이가 `var`를 쓰지 않는 이유 중 하나입니다.

- 대상: **같은 번호 비대면 week5** = JS 타입/변수/var-let-const/호이스팅/TDZ
- 구성:
  - 1-1. **핵심 리마인드** (구술 설명 옆에 두고 읽는 용도, 비유·코드 스니펫 중심):
    - `undefined` vs `null` — 엔진이 준 값 vs 개발자가 준 값 (기존 대면 코드 유지)
    - `var`의 함수 스코프 — `if` 블록을 무시하는 `var secret` 예시 (기존 유지)
    - 호이스팅 시각화 — `console.log(x); var x = 5;` 단계별 주석 (기존 유지)
    - **TDZ 시각화** (신규 강조 — 비대면 섹션 2-3 신규 보강과 맞물림): `let a; console.log(a); let a = 1;` 같은 2~3단계 코드. "선언은 호이스팅, 초기화는 선언문에서" 한 줄.
    - `NaN === NaN`이 `false`인 이유 (기존 유지)
  - 1-2. **사전학습 인증 미션 풀이** — week5 비대면 인증 미션 Q1~Q4 중 **Q1(typeof NaN)과 Q2(TDZ) 두 문제만 선별 풀이**. Q3(==/undefined)·Q4(falsy)는 **섹션 3 면접 질문**과 겹치므로 이 섹션에서는 풀이하지 않고 "면접 질문에서 다시 봅니다"로 넘김.
    - Q1 풀이: 정답 `"number"` + IEEE 754 배경 + `Number.isNaN()` 사용 권고 + 흔한 오답(`value !== value` 트릭은 의도 불명확)
    - Q2 풀이: 정답 `ReferenceError` + TDZ 정의 + "var는 호이스팅 시 undefined로 초기화, let/const는 선언문에서 초기화" 대비.
  - 1-3. **막힌 사람 클리닉 (상시)** — 비대면에서 막힌 포인트 자유 공유. "이해 안 됐어요"가 스터디의 기본값.

##### 3-2. 웹 개발 프로세스 5단계 — 프론트가 어디에 있는가

- 구성:
  - 5단계 간단 설명 (불릿 리스트):
    - **기획(PM/PO)** — 요구사항 정의, 와이어프레임
    - **디자인(디자이너)** — Figma로 UI 시안 제작
    - **프론트엔드(우리)** — 시안을 HTML/CSS/JS로 구현, 서버와 통신
    - **백엔드** — 데이터 저장/조회, API 제공
    - **배포·운영(DevOps)** — 서버 올리기, 에러 모니터링
  - "프론트엔드가 실제로 하는 일" 한 문단:
    - 디자인 시안 받아서 → HTML/CSS로 **화면 그리기** → JS로 **동작 붙이기** → 백엔드 API로 **데이터 주고받기** → 배포
  - 씨앗: week8에서 다룰 "통신·API"는 이 "데이터 주고받기"의 실제 동작, week10에서 다룰 "React 생태계"는 이 "화면 그리기 + 동작 붙이기"를 훨씬 편하게 만드는 도구.
- 코드 예시: 없음 (프로세스 설명이라 도식/리스트만).
- 연결 고리: "오늘 얘기한 '프론트 → 백엔드 통신' 부분은 week8에서 다시 만납니다."

##### 3-3. 프론트엔드 스택의 역사 (짧게) — 왜 React가 나왔나

- 구성 (시대별로 3단계):
  - **바닐라 JS + jQuery 시대 (~2010년대 초)**: "버튼 클릭하면 이 DOM을 직접 찾아서 바꿔라"를 손으로 썼음. 화면 복잡해지면 유지보수 지옥.
  - **React/Vue 등장 (2013~)**: "상태가 바뀌면 화면이 자동으로 다시 그려진다"는 발상. 선언형.
  - **Next.js/Vite 시대 (2020~)**: React 앱을 "그냥 만들면" 느리니까, SSR·번들링·라우팅을 기본으로 제공.
- 씨앗: week10에서 "React / Vite / Next.js"를 이름만 본격 소개. 지금은 "이 셋이 왜 나왔나" 맥락만 준다.
- 코드 예시: 없거나, 1줄 대비 (`document.getElementById('x').innerText = '...'` vs 선언적 `<div>{state}</div>` 주석 수준). 10줄 넘으면 안 됨.
- 연결 고리: "week10쯤 'Vite가 뭐예요?' 설명 들을 때 '아, 그때 그 얘기구나' 싶으면 성공."

##### 3-4. 취업 시장 스택 동향 (2026년 기준)

- 구성 (불릿 리스트 + 짧은 설명):
  - **메인 프레임워크**: React가 채용 공고의 약 70~80%. Vue는 국내 특정 회사(토스 일부, 우아한형제들 일부)에서. Svelte/Solid는 사이드.
  - **메타 프레임워크**: Next.js가 신규 프로젝트의 사실상 기본. Nuxt는 Vue 생태계에서.
  - **타입 시스템**: TypeScript는 이제 옵션이 아니라 기본. 신입 공고에도 "TS 경험 우대"가 "TS 필수"로 바뀌는 중.
  - **스타일링**: Tailwind CSS가 과반. CSS-in-JS(styled-components, emotion)는 레거시에서, 최신은 Vanilla Extract나 CSS Modules로 이동.
  - **상태관리**: Redux는 레거시, 신규는 Zustand / Jotai / TanStack Query가 주류.
  - **배포**: Vercel이 프론트 배포의 기본값. 백엔드와 함께 올릴 때는 AWS/GCP.
- 면접 맥락 한 줄: "공고를 볼 때 '이 회사가 어느 스택을 쓰는가'로 회사의 현대성을 가늠할 수 있습니다. 너무 오래된 스택만 있으면 레거시 유지보수 비중이 크다고 봅니다."
- 톤: 통계 숫자는 대략값으로. "면접 때 이거 물어봅니다" 한 줄만.
- 연결 고리: "week10에서 React/Next/Tailwind를, week11에서 Zustand/Jotai와 Vercel 배포를 다시 봅니다."
