<Subtitle level={2}>Series 1 — 모노레포 프론트엔드, 도커로 격리하여 배포하기</Subtitle>

<Paragraph>
목표는 단순했다. cs.kyonggi.ac.kr 학과 서버에 졸업논문 시스템을 포트 80으로 배포하는 것. pnpm 기반 모노레포 환경이었고, 교수님 지침으로 <strong>Reverse Proxy 사용 금지</strong>라는 제약이 붙었다. 각 서비스를 독립 포트로 직접 노출해야 했다.
</Paragraph>

<Paragraph>
처음엔 리버스프록시 없이 어떻게 하라는 건지 막막했는데, 덕분에 도커 네트워크와 포트 포워딩 구조를 훨씬 깊게 파고들 수 있었다. Multi-stage Build로 node:20-alpine에서 빌드하고 nginx:alpine에 결과물만 담았더니 이미지 크기가 대폭 줄었고, `-p 80:80`으로 호스트와 컨테이너 포트를 직접 연결했다.
</Paragraph>

<Paragraph>
여기서 확실히 배운 건 <strong>웹서버와 리버스프록시의 차이</strong>다. 웹서버는 정적 파일을 서빙하는 거고, 리버스프록시는 요청을 다른 포트로 넘기는 것. 이 두 가지가 머릿속에서 뒤섞여 있었는데, 제약 조건 덕분에 명확하게 분리됐다.
</Paragraph>

---

<Subtitle level={2}>Series 2 — GitHub Actions로 Push to Deploy 자동화</Subtitle>

<Paragraph>
매번 서버에 접속해서 git pull, build, restart 하는 게 너무 비효율적이었다. CI/CD를 직접 구축해보기로 했다.
</Paragraph>

<Paragraph>
ubuntu-latest Runner에서 이미지를 빌드하고 Docker Hub에 push하는 워크플로우를 작성했다. 교수님 가이드에 따라 docker-compose 대신 <strong>순수 docker run 명령어</strong>를 SSH로 서버에 전달하는 방식을 택했다. 트리거도 분리해서, PR 시엔 빌드 테스트만, 실제 Merge(Push) 시에만 서버 배포가 돌아가게 했다.
</Paragraph>

<Paragraph>
맥(로컬)에서 코드 작성, 우분투(Runner)에서 빌드, 리눅스(서버)에서 실행. 이 흐름을 직접 겪으면서 <strong>도커의 이식성</strong>을 체감했다. 그리고 코드를 수정하는 게 아니라 새 이미지로 갈아끼우는 Immutable Infrastructure 개념도 자연스럽게 익혔다.
</Paragraph>

---

<Subtitle level={2}>Series 3 — 프론트엔드 환경변수 주입</Subtitle>

<Paragraph>
로컬에서는 `localhost:8080`, 운영에서는 `cs.kyonggi.ac.kr`로 API 주소가 달라진다. 코드 수정 없이 이걸 관리하고 싶었다.
</Paragraph>

<Paragraph>
Vite의 `.env` 파일로 빌드 시점에 API 주소를 정적으로 주입하고, `|| 'localhost:8080'` 패턴으로 환경변수가 없을 때의 fallback을 보장했다. 프론트엔드는 <strong>빌드 시점에 변수가 고정</strong>된다는 특성(Static Replacement)이 있어서, 이미지를 구울 때 주소를 넣는 방식이 맞았다.
</Paragraph>

<Paragraph>
교수님의 "최소화" 요구사항은 결국 실행 시점에 환경변수를 주렁주렁 달지 않는 것이었다. 빌드 타임에 깔끔하게 넣고 끝.
</Paragraph>

---

<Subtitle level={2}>Series 4 — 죽지 않는 서비스를 위한 운영</Subtitle>

<Paragraph>
배포는 해봤으니, 이제 운영이었다. `docker rm -f`로 중지와 삭제를 원자적으로 처리해서 배포 다운타임을 최소화했고, `docker logs -f`로 컨테이너 내부 Nginx 상태를 실시간으로 추적했다. 항상 새 이미지를 실행하는 <strong>불변성 원칙</strong>을 적용해서 컨테이너 내부를 직접 수정하는 일은 없도록 했다.
</Paragraph>

<Paragraph>
배포보다 어려운 게 운영이라는 걸 체감했다. 서비스가 죽었을 때 어떻게 살릴지, 로그를 어떻게 추적할지 고민하는 과정이 오히려 더 큰 성장이었다.
</Paragraph>
