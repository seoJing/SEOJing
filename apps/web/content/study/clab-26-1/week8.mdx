<Subtitle level={2}>1. 클로저(Closure) — 렉시컬 환경을 기억하는 함수</Subtitle>

<Paragraph>
  먼저 한 줄 리마인드 —{" "}
  <strong>
    렉시컬 스코프는 "함수가 선언된 위치"를 기준으로 접근 가능한 상위 변수가
    정해지는 규칙
  </strong>
  이었어요(week6 복습). 클로저는 바로 이 렉시컬 환경을{" "}
  <strong>함수가 끝난 뒤에도 계속 기억</strong>하는 현상입니다. 렉시컬 스코프의
  자연스러운 결과물이고, 이 특성이 없으면 React의 `useState` 같은 것도 동작 못
  해요.
</Paragraph>

```js
function makeCounter() {
  let count = 0; // 외부 함수의 변수

  return function () {
    // 내부 함수 — 클로저
    count++;
    return count;
  };
}

const counter = makeCounter();
// makeCounter()는 이미 종료됐지만
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3
// count 변수가 여전히 살아있음
```

<Subtitle level={3}>데이터 은닉</Subtitle>

<Paragraph>
  클로저를 활용하면 외부에서 직접 접근할 수 없는 <strong>private 변수</strong>를
  만들 수 있습니다.
</Paragraph>

```js
function createAccount(initialBalance) {
  let balance = initialBalance; // 외부에서 직접 접근 불가

  return {
    deposit(amount) {
      balance += amount;
    },
    withdraw(amount) {
      if (amount > balance) return "잔액 부족";
      balance -= amount;
    },
    getBalance() {
      return balance;
    },
  };
}

const account = createAccount(1000);
account.deposit(500);
console.log(account.getBalance()); // 1500
console.log(account.balance); // undefined — 직접 접근 불가
```

<Subtitle level={2}>2. Promise — 콜백 지옥 탈출</Subtitle>

<Paragraph>
  서버에서 데이터를 가져오는 `fetch`, 타이머 `setTimeout` 등은 결과가 나중에
  옵니다. 과거에는 콜백으로 처리했고, 중첩이 깊어지면 <strong>콜백 지옥</strong>
  이 생겼습니다. Promise는 비동기 작업의 결과를 <strong>값(객체)</strong>으로
  표현하여 이를 해결합니다.
</Paragraph>

```js
// 콜백 지옥
fetch("/user", function (user) {
  fetch("/posts?userId=" + user.id, function (posts) {
    fetch("/comments?postId=" + posts[0].id, function (comments) {
      // 점점 깊어짐...
    });
  });
});

// Promise 체이닝 — 수평으로 나열
fetch("/user")
  .then((user) => fetch("/posts?userId=" + user.id))
  .then((posts) => fetch("/comments?postId=" + posts[0].id))
  .then((comments) => console.log(comments))
  .catch((err) => console.error(err));
```

<Subtitle level={3}>Promise 상태</Subtitle>

- <strong>`pending`</strong>: 비동기 작업 진행 중 (초기 상태)
- <strong>`fulfilled`</strong>: 작업 성공 — `.then()` 콜백 실행
- <strong>`rejected`</strong>: 작업 실패 — `.catch()` 콜백 실행
- 한 번 fulfilled/rejected 상태가 되면 변경 불가

<Subtitle level={3}>여러 Promise 동시 처리</Subtitle>

<Paragraph>
  아래 코드에 `await`가 미리 등장합니다.{" "}
  <strong>`await`는 3번 섹션에서 정식으로 설명합니다</strong> — 지금은
  "Promise가 완료될 때까지 기다렸다가 값을 꺼낸다" 정도로만 읽어주세요.
</Paragraph>

```js
// 순차 실행 — 느림 (앞이 끝나야 다음 시작)
async function sequential() {
  const a = await fetchA();
  const b = await fetchB();
  return [a, b];
}

// 병렬 실행 — 동시에 시작하고 모두 완료 기다림
async function parallel() {
  const [a, b] = await Promise.all([fetchA(), fetchB()]);
  return [a, b];
}

// 가장 먼저 완료된 것만 사용
async function race() {
  return await Promise.race([fetchA(), fetchB()]);
}
```

<Subtitle level={2}>3. async/await — Promise를 동기처럼</Subtitle>

<Paragraph>
  `async/await`은 Promise를 기반으로 한{" "}
  <strong>문법적 설탕(syntactic sugar)</strong>입니다. 비동기 코드를 마치 동기
  코드처럼 읽기 쉽게 작성할 수 있게 해줍니다.
</Paragraph>

```js
// Promise 체이닝
function getUser(id) {
  return fetch(`/users/${id}`)
    .then((res) => res.json())
    .then((user) => user)
    .catch((err) => {
      throw err;
    });
}

// async/await — 같은 동작, 더 직관적
async function getUser(id) {
  try {
    const res = await fetch(`/users/${id}`);
    const user = await res.json();
    return user;
  } catch (err) {
    throw err;
  }
}
```

<Subtitle level={3}>주의: await 병렬 실행</Subtitle>

```js
// 잘못된 예 — 순차 실행 (불필요하게 느림)
async function bad() {
  const a = await fetchA(); // fetchA 완료 대기
  const b = await fetchB(); // 그 다음에야 fetchB 시작
}

// 올바른 예 — 병렬 실행
async function good() {
  const [a, b] = await Promise.all([fetchA(), fetchB()]);
}
```

<Paragraph>
  실무에서는 React 컴포넌트에서 데이터를 가져올 때 `useEffect` 안에서 async
  함수를 정의하고 호출하는 패턴을 주로 사용합니다. 아래 코드는{" "}
  <strong>React 문법이라 지금 당장 이해 안 해도 됩니다</strong>. "async 함수를
  만들고 그 안에서 await를 쓴다" 패턴만 눈에 익혀두세요. React는 week10에서
  만납니다.
</Paragraph>

```js
useEffect(() => {
  async function fetchData() {
    try {
      const res = await fetch("/api/data");
      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err);
    }
  }

  fetchData();
}, []);
```

<Subtitle level={3}>fetch 실전 패턴 — 성공, 실패, 로딩을 나눠서 보기</Subtitle>

<Paragraph>
  `fetch()`를 사용할 때 가장 많이 하는 실수는 "응답이 왔으니 성공"이라고
  생각하는 것입니다. `fetch()`는 네트워크 자체가 실패한 경우에는 reject되지만,
  <strong>
    404나 500 같은 HTTP 에러 응답은 Promise reject로 처리하지 않습니다
  </strong>
  . 서버가 응답을 보내긴 했기 때문입니다. 그래서 `res.ok`나 `res.status`를 직접
  확인해야 합니다.
</Paragraph>

```js
async function fetchUser(userId) {
  const res = await fetch(`/api/users/${userId}`);

  if (!res.ok) {
    throw new Error(`HTTP error: ${res.status}`);
  }

  const data = await res.json();
  return data;
}
```

<Paragraph>
  화면에서는 보통 데이터를 가져오는 동안의 <strong>로딩 상태</strong>, 실패했을
  때의 <strong>에러 상태</strong>, 성공했을 때의 <strong>데이터 상태</strong>를
  나눠서 다룹니다. 아직 React를 배우기 전이라도 이 세 가지 이름은 꼭
  기억해두세요.
</Paragraph>

```js
let loading = true;
let error = null;
let data = null;

try {
  data = await fetchUser(1);
} catch (err) {
  error = err;
} finally {
  loading = false;
}
```

<Paragraph>
  요청끼리 서로 의존하지 않는다면 `Promise.all()`로 병렬 실행할 수 있습니다.
  예를 들어 사용자 정보와 알림 개수는 동시에 가져와도 됩니다. 반대로 두 번째
  요청이 첫 번째 요청 결과에 의존한다면 순차적으로 `await`해야 합니다.
</Paragraph>

```js
// 서로 독립적인 요청 — 병렬 가능
const [user, notifications] = await Promise.all([
  fetchUser(userId),
  fetchNotifications(userId),
]);

// 앞 요청 결과가 뒤 요청에 필요 — 순차 실행
const user = await fetchUser(userId);
const posts = await fetchPostsByUser(user.id);
```

<Paragraph>
  이 패턴들이 대면 8주차의 API 계약 이야기와 연결됩니다. 프론트엔드 개발자는
  단순히 요청을 보내는 것에서 끝나지 않고, 실패 응답이 어떤 형태로 오는지, 빈
  데이터는 어떻게 오는지, 어떤 요청을 병렬로 묶을 수 있는지까지 백엔드와 맞춰야
  합니다.
</Paragraph>
