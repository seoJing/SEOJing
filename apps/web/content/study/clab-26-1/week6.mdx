<Subtitle level={2}>1. JS의 타입 시스템 — 다른 언어와 뭐가 다른가</Subtitle>

<Paragraph>
  Java나 C++에서는 변수를 선언할 때 타입을 명시합니다.{" "}
  <strong>정적 타입 언어</strong>라고 부릅니다. 반면 JS는{" "}
  <strong>동적 타입 언어</strong>로, 변수에 어떤 타입의 값이든 넣을 수 있고
  런타임에 타입이 결정됩니다.
</Paragraph>

```js
// Java였다면: int x = 1;
// JS에서는
let x = 1; // 지금은 숫자
x = "hello"; // 이제 문자열 — 에러 없음
x = true; // 이제 불리언 — 에러 없음
x = null; // 이제 null — 에러 없음
```

<Paragraph>
  JS의 원시 타입은 7가지입니다:{" "}
  <strong>number, string, boolean, undefined, null, symbol, bigint</strong>. 이
  외에 참조 타입으로 <strong>object</strong>(배열, 함수, 일반 객체 포함)가
  있습니다.
</Paragraph>

<Subtitle level={3}>number 타입의 특이점</Subtitle>

<Paragraph>
  대부분의 언어는 정수와 실수 타입이 분리되어 있습니다(int, float, double 등).
  JS는
  <strong>number 하나</strong>로 모든 숫자를 표현합니다. 내부적으로 IEEE 754
  64비트 부동소수점을 사용합니다. 여기서 유명한 버그가 나옵니다.
</Paragraph>

```js
0.1 + 0.2 === 0.3; // false
0.1 + 0.2; // 0.30000000000000004
```

<Paragraph>
  이는 JS만의 문제가 아니라 부동소수점 표현 방식 자체의 한계입니다. 실무에서
  금액 계산 등 정밀도가 중요한 경우에는 정수로 변환하거나 라이브러리를 사용해야
  합니다.
</Paragraph>

<Paragraph>number 타입에는 세 가지 특별한 값도 있습니다.</Paragraph>

```js
Infinity - // 양의 무한대
  Infinity; // 음의 무한대
NaN; // Not a Number — 숫자가 아님을 나타내는 숫자(!)
```

```js
1 / 0 - // Infinity
  1 / 0; // -Infinity
"hello" * 2; // NaN
NaN === NaN; // false — NaN은 자기 자신과도 같지 않음
Number.isNaN(NaN); // true — NaN 확인은 이걸 써야 함
```

<Subtitle level={3}>undefined vs null — 둘 다 "없음"인데 왜 두 개?</Subtitle>

<Paragraph>
  다른 언어에서는 보통 null 하나로 "값 없음"을 표현하는데, JS에는 undefined와
  null이 둘 다 있습니다.
</Paragraph>

- <strong>undefined</strong>: JS 엔진이 자동으로 할당. "변수는 있는데 아직 값이
  없다"
- <strong>null</strong>: 개발자가 의도적으로 할당. "이 변수는 비어있다고
  명시한다"

```js
let x; // undefined — 엔진이 자동 부여
let y = null; // null — 개발자가 의도적으로 "비어있음" 명시
```

<Paragraph>
  실무에서 권장하는 방식은{" "}
  <strong>변수를 의도적으로 비울 때는 null을 사용</strong>하고, undefined는 직접
  할당하지 않는 것입니다. undefined를 직접 할당하면 "아직 초기화 안 됨"인지
  "의도적으로 비운 것"인지 구분이 안 되기 때문입니다.
</Paragraph>

<Subtitle level={2}>
  2. var / let / const — 어떻게 다르고 왜 var를 쓰지 말아야 하나
</Subtitle>

<Paragraph>
  다른 언어 경험자라면 변수 선언 자체는 낯설지 않을 텐데, JS의 var는 상당히
  특이합니다. ES6(2015년)에서 let과 const가 나온 이유가 있습니다.
</Paragraph>

<Subtitle level={3}>var의 문제점 1: 함수 스코프</Subtitle>

```js
// Java, Python이라면 블록 밖에서 x를 쓸 수 없음
// JS var는 다릅니다
if (true) {
  var x = 10;
}
console.log(x); // 10 — 블록 밖에서도 접근 가능!

// let은 예상대로 동작
if (true) {
  let y = 10;
}
console.log(y); // ReferenceError: y is not defined
```

<Subtitle level={3}>var의 문제점 2: 호이스팅(Hoisting)</Subtitle>

<Paragraph>
  JS 엔진은 코드를 실행하기 전에 var 선언을 스코프 최상단으로
  끌어올립니다(호이스팅). 선언은 올라가지만 값 할당은 올라가지 않습니다.
</Paragraph>

```js
console.log(x); // undefined — 에러가 아님!
var x = 5;
console.log(x); // 5

// JS 엔진이 실제로 실행하는 방식
var x; // 선언이 여기로 끌어올려짐
console.log(x); // undefined
x = 5; // 할당은 원래 위치에서 실행
console.log(x); // 5
```

<Paragraph>
  let/const도 호이스팅은 일어나지만, 선언 전에 접근하면{" "}
  <strong>TDZ(Temporal Dead Zone)</strong>에 의해 ReferenceError가 발생합니다.
  이 동작이 훨씬 예측 가능하고 안전합니다.
</Paragraph>

<Subtitle level={3}>var의 문제점 3: 중복 선언 허용</Subtitle>

```js
var x = 1;
var x = 2; // 에러 없음, 그냥 덮어씀
console.log(x); // 2

let y = 1;
let y = 2; // SyntaxError: Identifier 'y' has already been declared
```

<Subtitle level={3}>결론: 언제 뭘 쓸까</Subtitle>

|                  | var       | let                  | const                |
| :--------------- | :-------- | :------------------- | :------------------- |
| 스코프           | 함수      | 블록                 | 블록                 |
| 재선언           | 가능      | 불가                 | 불가                 |
| 재할당           | 가능      | 가능                 | 불가                 |
| 호이스팅 시 동작 | undefined | ReferenceError (TDZ) | ReferenceError (TDZ) |

<Paragraph>
  실무에서는 <strong>기본적으로 const</strong>를 쓰고, 재할당이 필요한 경우에만
  let을 씁니다. var는 쓰지 않습니다.
</Paragraph>

<Subtitle level={2}>
  3. 암묵적 타입 변환 — JS가 "이상하다"는 말이 나오는 이유
</Subtitle>

<Paragraph>
  JS는 연산 시 타입이 맞지 않으면 에러를 내는 대신 알아서 타입을 변환하려고
  시도합니다. 이를 <strong>암묵적 타입 변환(Implicit Type Coercion)</strong>
  이라고 합니다.
</Paragraph>

```js
// + 연산자: 문자열이 있으면 문자열 연결 우선
"5" + 1; // "51"
1 + "5"; // "15"
1 + 2 + "3"; // "33" — 왼쪽부터 평가: 3 + "3" = "33"
"3" + 1 + 2; // "312" — "3" + 1 = "31", "31" + 2 = "312"

// -, *, / 연산자: 숫자 변환 시도
"5" - 1; // 4
"5" * "2"; // 10
"5" - "3"; // 2
"hello" - 1; // NaN
```

<Paragraph>
  이 동작 때문에 JS를 처음 접하는 다른 언어 개발자들이 당황하는 경우가 많습니다.
  유명한 예시들을 직접 실행해보세요.
</Paragraph>

```js
[] + []      // ""
[] + {}      // "[object Object]"
{} + []      // 0 (브라우저 콘솔에서는 다를 수 있음)
true + true  // 2
false + 1    // 1
null + 1     // 1
undefined + 1 // NaN
```

<Subtitle level={3}>== 를 쓰면 안 되는 이유</Subtitle>

<Paragraph>
  ==는 비교 전에 타입을 맞추려고 암묵적 변환을 수행합니다. 결과가 직관적이지
  않습니다.
</Paragraph>

```js
0 == false; // true
0 == ""; // true
"" == false; // true
null == undefined; // true
null == 0; // false  ← 이상하지 않나요?
NaN == NaN; // false  ← NaN은 자신과도 다름
```

<Paragraph>
  ===는 타입 변환 없이 값과 타입이 모두 같아야 true를 반환합니다.
  <strong>항상 === 를 사용</strong>하는 것이 버그를 예방하는 기본입니다.
</Paragraph>

<Subtitle level={2}>4. 단축 평가 — 실무에서 자주 쓰는 JS 패턴</Subtitle>

<Paragraph>
  논리 연산자 ||와 &&는 boolean이 아닌 값을 반환한다는 점이 독특합니다. 이를
  활용한 패턴이 실무에서 자주 등장합니다.
</Paragraph>

```js
// || : 왼쪽이 falsy면 오른쪽 반환, truthy면 왼쪽 반환
"hello" || "default"; // "hello"
"" || "default"; // "default"
null || "default"; // "default"
0 || 42; // 42

// && : 왼쪽이 falsy면 왼쪽 반환, truthy면 오른쪽 반환
"hello" && "world"; // "world"
"" && "world"; // ""
null && "world"; // null
```

<Paragraph>React 코드에서 이런 패턴을 자주 보게 됩니다.</Paragraph>

```js
// 기본값 설정
const name = userInput || "익명";

// 조건부 렌더링 (React에서 흔한 패턴)
isLoggedIn && <Dashboard />;

// null 병합 연산자 (??) — 0이나 ''도 유효한 값일 때 사용
const count = userCount ?? 0; // null, undefined일 때만 0으로 대체
// || 와 달리 0, ''는 그대로 유지됨
```
