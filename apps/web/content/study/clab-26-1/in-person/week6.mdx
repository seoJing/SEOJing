<Subtitle level={3}>CSS Grid</Subtitle>

- <strong>Flexbox</strong>: 1차원 (가로 또는 세로 한 방향)
- <strong>Grid</strong>: 2차원 (행과 열 동시 제어)
- <strong>`1fr`</strong>: 남은 공간을 비율로 나눔
- <strong>`repeat(auto-fill, minmax(280px, 1fr))`</strong>: 미디어 쿼리 없이 자동 반응형

<Subtitle level={3}>미디어 쿼리</Subtitle>

- <strong>모바일 퍼스트</strong>: 기본 스타일을 모바일로, `min-width`로 큰 화면 대응
- <strong>`prefers-color-scheme`</strong>: 다크 모드 감지
- <strong>`prefers-reduced-motion`</strong>: 접근성, 애니메이션 줄이기

<Subtitle level={2}>2. 저번 주 과제 정답 및 풀이</Subtitle>

<Subtitle level={3}>학습 자료 5주차 과제</Subtitle>

<Paragraph>
  <strong>
    Q1. `null`과 `undefined`의 차이를 설명해주세요.
  </strong>
</Paragraph>

<Paragraph>
  둘 다 "값이 없음"을 나타내지만 주체가 다릅니다.{" "}
  <strong>undefined</strong>는 JS 엔진이 자동으로 할당하는 값으로, 변수를 선언만 하고
  값을 넣지 않았을 때 자동으로 들어갑니다. <strong>null</strong>은 개발자가 의도적으로
  "이 변수는 비어있다"고 명시할 때 사용합니다. 실무에서는 변수를 의도적으로 비울 때
  null을 쓰고, undefined는 직접 할당하지 않는 것이 권장됩니다.
</Paragraph>

<Paragraph>
  <strong>
    Q2. `==`와 `===`의 차이는 무엇인가요?
  </strong>
</Paragraph>

<Paragraph>
  <strong>`==`</strong>는 비교 전에 타입을 자동으로 변환(암묵적 강제 변환)한 후 비교합니다.
  그래서 `0 == false`가 `true`가 되는 등 직관적이지 않은 결과가 나옵니다.{" "}
  <strong>`===`</strong>는 타입 변환 없이 값과 타입이 모두 같아야 `true`를 반환합니다.
  예상치 못한 버그를 방지하기 위해 실무에서는 항상 `===`를 사용합니다.
</Paragraph>

<Subtitle level={2}>4. 면접 질문</Subtitle>

<Paragraph>
  이번 주 범위와 관련된 실제 면접 질문들입니다. 정답을 외우기보다는 자신만의 말로

</Paragraph>

<Subtitle level={3}>Q1. `null`, `undefined`, undeclared(선언되지 않은 변수)의 차이는 무엇인가요?</Subtitle>

- <strong>`undefined`</strong>: 변수가 선언됐지만 값이 할당되지 않은 상태. JS 엔진이 자동 부여.
- <strong>`null`</strong>: 개발자가 의도적으로 "값 없음"을 명시한 상태. `typeof null`은 버그로 인해 `'object'`를 반환.
- <strong>undeclared</strong>: 선언 자체가 없는 변수. `typeof`로 접근하면 `'undefined'`를 반환하지만, 직접 참조하면 `ReferenceError` 발생.

```js
let a;
console.log(a);          // undefined
console.log(typeof a);   // 'undefined'

let b = null;
console.log(b);          // null
console.log(typeof b);   // 'object' ← 버그

console.log(typeof c);   // 'undefined' (선언 안 했지만 typeof는 안전)
console.log(c);          // ReferenceError: c is not defined
```

<Subtitle level={3}>Q2. `var`, `let`, `const`의 차이를 설명해주세요.</Subtitle>

<Paragraph>
  세 가지 모두 변수 선언 키워드지만 스코프, 재선언, 재할당, 호이스팅 동작이 다릅니다.
</Paragraph>

- <strong>`var`</strong>: 함수 스코프, 재선언 가능, 호이스팅 시 `undefined`로 초기화. 예측하기 어려워 현재는 사용하지 않습니다.
- <strong>`let`</strong>: 블록 스코프, 재선언 불가, 재할당 가능. TDZ로 인해 선언 전 접근 시 `ReferenceError`.
- <strong>`const`</strong>: 블록 스코프, 재선언·재할당 모두 불가. 단, 객체·배열의 내부 값은 변경 가능.

<Paragraph>
  면접 팁: "기본적으로 const를 쓰고, 재할당이 필요한 경우에만 let을 쓴다. var는 쓰지 않는다"고
  정리하면 좋습니다.
</Paragraph>

<Subtitle level={3}>Q3. `==`와 `===`의 차이는 무엇인가요?</Subtitle>

<Paragraph>
  <strong>`==`(동등 연산자)</strong>는 비교 전에 두 값의 타입을 맞추는 암묵적 타입 변환을
  수행합니다. 그래서 `0 == false`, `'' == false`, `null == undefined`가 모두 `true`가 됩니다.
  <strong>`===`(일치 연산자)</strong>는 타입 변환 없이 값과 타입이 모두 같아야 `true`를 반환합니다.
  예측 불가능한 버그를 방지하기 위해 실무에서는 항상 `===`를 사용합니다.
</Paragraph>
