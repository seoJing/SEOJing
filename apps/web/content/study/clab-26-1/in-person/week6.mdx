<Subtitle level={2}>1. 이번 주 사전학습 점검</Subtitle>

<Paragraph>
  이번 주 사전학습(비대면 week6)에서는 <strong>객체</strong>,{" "}
  <strong>함수(일급 객체)</strong>,<strong>스코프(함수/블록/렉시컬)</strong> 세
  축을 다뤘습니다. JS를 이해하는 심장 같은 개념들이라, 오늘 대면 앞부분에서 구술
  설명용으로 다시 짚고 넘어갑니다. 완전한 이해가 목표가 아니라, 코드를 옆에 두고
  말로 풀어낼 수 있는 상태를 만드는 것이 목표입니다.
</Paragraph>

<Subtitle level={3}>핵심 리마인드</Subtitle>

<Subtitle level={4}>참조 복사의 함정 — 같은 주소를 두 변수가 가리킨다</Subtitle>

<Paragraph>
  원시값(숫자·문자열·불리언 등)은 <strong>값 자체가 복사</strong>되지만, 객체는{" "}
  <strong>주소(참조)가 복사</strong>됩니다. 같은 객체를 두 변수가 바라보게 되니,
  한쪽을 바꾸면 다른 쪽도 같이 바뀝니다. 이게 이번 주 이후로 계속 따라다닐
  감각입니다.
</Paragraph>

```js
// 원시값 — 값이 복사됨. 독립적.
let a = 1;
let b = a;
b = 99;
console.log(a); // 1 — 영향 없음

// 객체 — 주소가 복사됨. 같은 객체를 가리킴.
const obj1 = { x: 1 };
const obj2 = obj1;
obj2.x = 99;
console.log(obj1.x); // 99 — 같이 바뀜
```

<Paragraph>
  이 감각이 다음 주(week7)의 <strong>불변성</strong> 주제로 바로 이어집니다.
  "`const`로 선언했는데 왜 내부 값이 바뀌지?"의 답이 여기에 있습니다.
</Paragraph>

<Subtitle level={4}>
  함수 선언식 vs 표현식 — 호이스팅은 됐지만 값이 없는 상태
</Subtitle>

<Paragraph>
  함수 선언식은 <strong>선언과 값이 함께</strong> 호이스팅되지만, 함수
  표현식(`const fn = function() {}`)은
  <strong>변수 선언만</strong> 호이스팅됩니다. 선언 전에 호출하면 에러 종류가
  달라지는 이유가 여기에 있습니다.
</Paragraph>

```js
foo(); // OK — 선언식은 전체가 올라감
bar(); // TypeError — var 표현식은 undefined 상태
baz(); // ReferenceError — const 표현식은 TDZ

function foo() {}
var bar = function () {};
const baz = function () {};
```

<Paragraph>
  포인트는 "호이스팅이 안 돼서 에러"가 아니라{" "}
  <strong>"호이스팅은 됐는데 값이 없는 상태에서 호출해서 에러"</strong>라는
  것입니다. week5에서 다룬 TDZ 감각이 여기서 다시 쓰입니다.
</Paragraph>

<Subtitle level={4}>일급 객체가 왜 중요한가 — React와 연결</Subtitle>

<Paragraph>
  함수가 일급 객체라는 것은 <strong>함수를 값처럼 다룰 수 있다</strong>는
  뜻입니다. 변수에 담고, 인자로 넘기고, 반환값으로 쓸 수 있다는 것. 이게 왜
  중요한지 React 코드 한 줄로 확인해봅시다.
</Paragraph>

```jsx
// React에서 이벤트 핸들러 — handleClick 함수 자체를 값으로 넘김
<button onClick={handleClick}>클릭</button>
```

<Paragraph>
  `onClick={handleClick}`에서 `handleClick`은 "함수를 호출한 결과"가 아니라{" "}
  <strong>"함수 자체"</strong>입니다. 이게 가능한 이유는 JS에서 함수가 일급
  객체이기 때문. week10 React 주차에서 "왜 괄호를 안 붙여요?" 질문이 반드시
  나오는데, 오늘 이 문장이 그 답입니다.
</Paragraph>

<Subtitle level={4}>렉시컬 스코프 — 함수는 작성된 곳의 규칙을 따른다</Subtitle>

<Paragraph>
  함수의 상위 스코프는 <strong>호출된 위치</strong>가 아니라{" "}
  <strong>선언된(작성된) 위치</strong>로 정해집니다. 전학생 비유가 제일
  직관적입니다 — 학생이 다른 반에 놀러 가도(호출 위치), 소속 반(선언 위치)의
  규칙을 따릅니다.
</Paragraph>

```js
const x = "global";

function foo() {
  console.log(x); // foo가 선언된 곳(전역)의 x를 참조
}

function bar() {
  const x = "bar";
  foo(); // bar 안에서 호출해도...
}

bar(); // "global" 출력
```

<Paragraph>
  이 감각이 몇 주 뒤 <strong>클로저</strong>(week11)의 기반이 됩니다. 지금은
  "선언된 곳이 기준" 한 줄만 가져가도 충분합니다.
</Paragraph>

<Subtitle level={3}>사전학습 인증 미션 풀이</Subtitle>

<Paragraph>
  인증 미션 Q1~Q4 중 오늘 대면에서는 <strong>Q1(호이스팅 TypeError)</strong>와{" "}
  <strong>Q2(렉시컬 스코프)</strong>만 함께 풀어봅니다. Q3(참조 복사)는 위 "참조
  복사의 함정"에서 이미 다뤘고, Q4(IIFE)는 오늘 대면 본론과 맥락이 머니{" "}
  <strong>비대면 자료를 참고해 정리한 답을 댓글로 제출</strong>해주세요. 궁금한
  점은 아래 "막힌 사람 클리닉"에서 바로 질문해도 됩니다.
</Paragraph>

<Paragraph>
  <strong>
    Q1. 함수 선언식은 호이스팅되어 선언 전에 호출할 수 있습니다. 그런데 함수
    표현식(`const fn = function() {}`)은 선언 전에 호출하면 `TypeError`가
    발생합니다. `ReferenceError`가 아니라 왜 `TypeError`인지, 호이스팅 과정을
    단계별로 설명하세요.
  </strong>
</Paragraph>

<Subtitle level={4}>정답</Subtitle>

<Paragraph>
  함수 선언식은 선언과 초기화가 함께 호이스팅되어 함수 전체가 올라갑니다. 반면{" "}
  <strong>
    `var` 함수 표현식은 변수 선언만 호이스팅되고 값(함수)은 `undefined`
  </strong>{" "}
  상태입니다. `undefined()`를 호출하면 "undefined is not a function"이라는{" "}
  <strong>`TypeError`</strong>가 발생합니다. `const`/`let` 표현식은 TDZ 때문에{" "}
  `ReferenceError`가 먼저 납니다.
</Paragraph>

<Subtitle level={4}>흔한 오답</Subtitle>

<Paragraph>
  "호이스팅이 안 돼서 에러가 난다"고 답하는 경우가 많습니다.{" "}
  <strong>
    호이스팅은 일어났지만, 값이 `undefined`인 상태에서 호출해서 나는 에러
  </strong>
  라는 점이 핵심입니다. 에러 타입 이름(`TypeError` vs `ReferenceError`)은 엔진이
  그 시점에 변수를 어떻게 인식하고 있는지를 드러내는 신호라고 보면 됩니다.
</Paragraph>

---

<Paragraph>
  <strong>
    Q2. 아래 코드의 실행 결과를 예측하고, 렉시컬 스코프 개념을 사용해 이유를
    설명하세요.
  </strong>
</Paragraph>

```js
const x = "global";

function foo() {
  console.log(x);
}

function bar() {
  const x = "bar";
  foo();
}

bar();
```

<Subtitle level={4}>정답: `"global"`</Subtitle>

<Paragraph>
  JS는 <strong>렉시컬 스코프(정적 스코프)</strong>를 사용합니다. 함수의 상위
  스코프는 "호출 위치"가 아니라 <strong>"선언 위치"</strong>에 의해 결정됩니다.
  `foo`는 전역에서 선언됐으므로 어디서 호출되든 전역 스코프의 `x`를 참조합니다.
</Paragraph>

<Subtitle level={4}>흔한 오답</Subtitle>

<Paragraph>
  `bar()` 내부에서 호출했으니 `bar`의 `x`인 `"bar"`가 출력된다고 예측하는 경우가
  많습니다. 이는 <strong>동적 스코프</strong> 방식의 사고인데, JS는 그렇게
  동작하지 않습니다. 호출 위치가 아니라 <strong>선언 위치</strong>가 기준이라는
  점이 핵심입니다.
</Paragraph>

<Paragraph>

  충분해요.
</Paragraph>

<Subtitle level={2}>
  2. [대면 메인] 팀 프로젝트 컨벤션·그라운드룰 + Git/PR 기초
</Subtitle>

<Paragraph>
  오늘 본론은 코드 문법이 아니라 <strong>사람</strong> 얘기입니다. 비대면에서 막
  배운 함수·변수·프로퍼티 이름 짓기가, 혼자 쓸 때와 팀으로 쓸 때 어떻게 의미가
  달라지는지. 그리고 Git/PR이라는 도구가 왜 생겼고, 그 안에서 리뷰 문화는 어떻게
  굴러가는지. 스터디 후반부(week10~11) 팀 프로젝트가 본격화되기 전에, 지금{" "}
  <strong>감각을 예습</strong>해두는 시간입니다.
</Paragraph>

<Subtitle level={3}>2-1. 왜 이 얘기를 지금 하는가</Subtitle>

<Paragraph>
  이번 주 비대면에서 함수·변수·프로퍼티 이름을 짓는 법을 막 배웠습니다. 오늘은
  그 "이름"이{" "}
  <strong>혼자 쓸 때와 팀으로 쓸 때 어떻게 의미가 달라지는지</strong>를 봅니다.
  코드 문법이 막 쌓이는 지금이 협업 감각을 심기에 가장 좋은 타이밍이라서요.
</Paragraph>

<Paragraph>
  오늘 미리 볼 키워드는 다섯 개입니다. 외울 필요는 없고, 이름과 윤곽만 기억하면
  충분합니다.
</Paragraph>

- <strong>팀으로 코드 쓰기</strong> — 왜 혼자 쓰는 것과 다른가, 왜 컨벤션이
  필요한가
- <strong>네이밍 컨벤션</strong> — 함수·변수 이름이 팀 속도를 정한다
- <strong>Git 기초 감각</strong> — commit·branch·PR이 왜 필요한가 (명령어 아님)
- <strong>PR 리뷰 문화</strong> — 리뷰는 트집이 아니라 품질 합의
- <strong>그라운드룰 · AI와 팀</strong> — 실제 팀이 어떻게 운영되는가, AI가 쓴
  코드를 올릴 때의 매너
- <strong>연결 지도</strong> — 오늘 미리 본 키워드가 앞으로 어느 주차에서
  본편으로 이어지는가

<Paragraph>
  이 시점에 이 얘기를 꺼내는 이유는 분명합니다. 팀 프로젝트가 시작되는 순간{" "}
  <strong>"Git이 어려워서 기능 구현을 시작도 못 하는" 사람</strong>이 반드시
  생깁니다. 지금 미리 짚어두면 그때 충격이 덜합니다. 오늘 전부 이해 못 해도
  됩니다. 이 단어들이 몇 주 뒤 팀플에서 다시 올라올 때 "아, 그때 그 얘기!"
  싶으면 오늘 역할은 다한 거예요.
</Paragraph>

<Subtitle level={3}>2-2. 혼자 vs 팀 — 코드가 커지면 뭐가 달라지는가</Subtitle>

<Paragraph>
  혼자 쓰는 코드는 간단합니다. 내가 쓴 이름·스타일·순서를 내가 이해하면 됩니다.
  <strong>내일의 내가 못 알아보는 수준으로만 안 쓰면</strong> 그럭저럭 굴러가요.
  변수 이름을 `x1`, `x2`로 지어도 "오늘의 나"는 안 헷갈립니다.
</Paragraph>

<Paragraph>
  팀 코드는 전제가 완전히 다릅니다. 핵심 전제를 한 문장으로 말하면:{" "}
  <strong>"3개월 뒤의 다른 사람(또는 3개월 뒤의 나)이 이 코드를 읽는다"</strong>
  입니다. 내 머릿속 맥락이 하나도 없는 사람이 읽고, 바로 이해할 수 있어야
  합니다. 안 그러면 팀원이 내 코드 앞에서 매번 10초씩 멈추고, 그 10초가 쌓이면
  팀 속도가 무너집니다.
</Paragraph>

<Paragraph>
  비유로 풀면 <strong>"혼자 쓰는 일기장"</strong>과{" "}
  <strong>"여러 명이 같이 쓰는 위키"</strong>의 차이입니다. 일기장은 나만 알면
  됩니다. 위키는 제목·카테고리·작성 규칙이 필요해요. 규칙이 없으면 다른 사람이
  검색해도 찾지 못하고, 새로 들어온 사람이 "어디서부터 읽어야 돼요?"라는 질문을
  던집니다.
</Paragraph>

<Paragraph>
  실무 감각으로 덧붙이자면, 주니어가 팀에 입사해서 가장 먼저 부딪히는 벽이{" "}
  <strong>"이 팀의 코드 스타일"</strong>입니다. 문법은 이미 알고 들어가는데, "이
  팀은 함수 이름을 이렇게 짓네", "여기서는 들여쓰기를 이렇게 하네" 같은 스타일
  차이가 먼저 부딪혀요. 그래서 컨벤션이라는 게 "개인 취향"이 아니라{" "}
  <strong>"팀이 합의한 공통어"</strong>로 작동합니다.
</Paragraph>

<Paragraph>
  이 "다른 사람이 내 코드를 읽는다"는 관점은 앞으로 두 번 더 크게 올라옵니다.
  week9 대면에서 다룰 <strong>보안</strong> 주제에서는{" "}
  <strong>"다른 사람(공격자)이 내 코드와 데이터를 만진다"</strong>는 관점으로
  확장되고, 팀 프로젝트 시점에는 <strong>"다른 팀원이 내 PR을 리뷰한다"</strong>
  는 관점으로 구체화됩니다. 오늘 심는 감각이 그때마다 다시 쓰입니다.
</Paragraph>

<Subtitle level={3}>
  2-3. 네이밍 컨벤션 — 함수·변수 이름이 팀 속도를 정한다
</Subtitle>

<Paragraph>
  비대면 week6에서 배운 함수·변수·프로퍼티 이름 짓기가, 혼자 쓸 때는{" "}
  <strong>"에러만 안 나면 OK"</strong>였지만 팀에서는{" "}
  <strong>"읽는 속도가 곧 개발 속도"</strong>가 됩니다. 구체적인 감각 몇 개를
  규칙 형태로 심어둘게요. 외우는 게 아니라 "이런 결이 있다"만 알아두세요.
</Paragraph>

- <strong>불리언은 `is` / `has` / `can`으로 시작</strong> — `loading` 대신
  `isLoading`, `admin` 대신 `isAdmin`. React 코드에서 매일 보게 될 패턴입니다.
- <strong>함수는 동사로 시작</strong> — `user()`는 뭘 하는 건지 모르지만,
  `getUser()`는 가져온다는 의도가 드러납니다. React에서는 이벤트 핸들러를
  `handleXxx`로 붙이는 관례가 있어요.
- <strong>단위는 이름에 표시</strong> — `timeout` 대신 `timeoutMs`, `width` 대신
  `widthPx`. 단위가 섞여서 생기는 버그를 이름으로 예방합니다.
- <strong>약어 남용 금지</strong> — `usrLstCnt` 대신 `userListCount`. 타이핑 3초
  아끼려다 읽는 사람이 30초씩 멈춥니다.

<Paragraph>
  규칙만 나열하면 추상적이니 10줄짜리 비교 예시로 감을 잡아봅시다.
</Paragraph>

```js
// Before — 팀원이 10초 동안 멈춰야 함
function check(u, t) {
  if (u.a > t) return true;
  return false;
}

// After — 읽자마자 이해됨
function isUserOlderThan(user, ageThreshold) {
  return user.age > ageThreshold;
}
```

<Paragraph>
  두 함수는 똑같이 동작합니다. 그런데 리뷰어 입장에서 Before는 "`u`가 뭐야,
  `a`가 뭐야, `t`가 뭐야"를 하나씩 추적해야 하고, After는 이름만 보고도 의도가
  잡힙니다. <strong>좋은 이름은 주석을 대체</strong>한다는 말이 여기서 나와요.
</Paragraph>

<Paragraph>
  AI 맥락으로 한 줄만 덧붙이면, 네이밍은 <strong>AI 시대에 더 중요</strong>
  해졌습니다. AI가 써준 코드의 변수명이 우리 팀 스타일과 다를 때, 그걸{" "}
  <strong>팀 스타일로 다시 이름 붙이는 일</strong>이 주니어의 핵심 업무 중
  하나입니다. 이름을 못 짓는다는 건 AI 코드를 팀에 못 녹인다는 뜻이기도 해요.
</Paragraph>

<Paragraph>
  오늘 얘기한 네이밍 감각은 <strong>week10</strong>에서 React 컴포넌트·훅 이름
  짓기로 다시 올라옵니다. 컴포넌트는 PascalCase(`LoginForm`), 훅은 `use`로
  시작(`useAuth`)이라는 규칙이 그때 등장해요. 지금은 "이런 결이 있다"만.
</Paragraph>

<Subtitle level={3}>
  2-4. Git 기초 감각 — commit · branch · PR이 왜 필요한가
</Subtitle>

<Paragraph>
  먼저 오늘 <strong>하지 않을 것</strong>부터 공표할게요. 오늘은{" "}
  <strong>Git 명령어를 외우지 않습니다</strong>. `git add`·`git commit`·`git
  push` 같은 명령어는 검색 한 번이면 나옵니다. 오늘은 "commit·branch·PR이 뭐
  하는 건지, 왜 필요한지"만 감으로 심어둡니다. 명령어 실습은 팀 프로젝트
  시작하는 시점에 따로 합니다.
</Paragraph>

<Subtitle level={4}>commit — "세이브 포인트"</Subtitle>

<Paragraph>
  게임에서 세이브 안 하고 보스 만나면 망하잖아요. Git의 commit이 정확히 그
  세이브입니다. 작업 중간중간 <strong>"여기까지는 확실히 동작함"</strong>을
  찍어두는 것.
</Paragraph>

<Paragraph>
  나중에 코드를 망쳐도 그 시점으로 돌아갈 수 있고, 다른 사람도 "언제 이 기능이
  들어갔지?"를 commit 기록으로 추적할 수 있습니다. commit 하나가 곧{" "}
  <strong>"어떤 변경이 왜 일어났는지"의 최소 단위</strong>예요.
</Paragraph>

<Subtitle level={4}>branch — "평행우주"</Subtitle>

<Paragraph>
  `main`은 원본 우주입니다. 내가 새 기능을 만들 때 원본을 직접 건드리면, 버그가
  생길 때 다 같이 망해요. 그래서 Git은{" "}
  <strong>
    "평행우주를 하나 떼어내서 거기서 작업하고, 다 되면 원본에 합치자"
  </strong>
  는 방식을 씁니다. 이 평행우주가 branch예요.
</Paragraph>

<Paragraph>
  팀원 셋이 동시에 다른 기능을 만들 때, 각자 자기 branch에서 작업합니다. 내가
  실수해도 다른 사람의 branch나 `main`은 안 망가져요. 완성된 기능만 원본에
  합칩니다. 그 "합치자"는 제안이 다음의 PR입니다.
</Paragraph>

<Subtitle level={4}>PR(Pull Request) — "합쳐달라는 제안서"</Subtitle>

<Paragraph>
  내 평행우주에서 완성한 기능을 `main`에 합치기 전에,{" "}
  <strong>"팀원들이 리뷰하고 합의해주세요"</strong>라고 올리는 문서가 PR입니다.
  GitHub/GitLab 같은 도구가 이걸 UI로 만들어줘요. PR 하나에는 "뭘 왜 바꿨는지"에
  대한 설명, 변경된 코드 파일 목록, 리뷰 코멘트가 붙습니다.
</Paragraph>

<Paragraph>
  왜 이런 게 필요한가? 혼자 쓰는 프로젝트라면 내 파일에 바로 저장해도 됩니다.
  근데 팀 3명이 같은 파일을 동시에 만지면{" "}
  <strong>충돌(두 사람이 같은 파일의 같은 줄을 동시에 다르게 바꾼 상황)</strong>
  이 나고, 누가 언제 왜 뭘 바꿨는지 흔적이 사라져요. Git은{" "}
  <strong>
    "누가 언제 왜 뭘 바꿨는지"를 전부 기록하고, 충돌을 드러내서 팀이 합의할 수
    있게 해주는 도구
  </strong>
  입니다. PR은 그 합의 자리예요.
</Paragraph>

<Paragraph>
  오늘 심은 commit/branch/PR 감각은 <strong>week11</strong>에서 다시 만납니다.
  Vercel이라는 배포 도구가 <strong>GitHub와 연결</strong>되어{" "}
  <strong>"main 브랜치에 머지되면 자동 배포"</strong>되는 흐름이 거기서 나와요.
  오늘의 "PR 머지 → main 갱신" 감각이 그때 "main 갱신 → 자동 배포"로 이어집니다.
</Paragraph>

<Subtitle level={3}>
  2-5. PR 리뷰 문화 — 리뷰는 트집이 아니라 품질 합의
</Subtitle>

<Paragraph>
  PR이 기술적 "합치기"만 하는 건 아니에요.{" "}
  <strong>"이 코드 괜찮은지 팀원들이 같이 보는 자리"</strong>이기도 합니다.
  그래서 리뷰 문화라는 게 팀 분위기를 크게 좌우해요. 주니어로 팀에 들어가서 가장
  먼저 배우는 협업 감각이 바로 이 리뷰 주고받기입니다.
</Paragraph>

<Subtitle level={4}>리뷰는 공격이 아니다</Subtitle>

<Paragraph>
  "이 코드 왜 이렇게 짰어요?"는 트집이 아니라 <strong>질문</strong>입니다.
  작성자가 "왜"를 설명하다 보면, 본인 코드를 다시 한 번 보게 돼요. 그 과정에서{" "}
  <strong>리뷰어도 배우고, 작성자도 배웁니다</strong>. 리뷰어가 "이게 틀렸다"가
  아니라 "내가 놓친 맥락이 있나?"라는 자세로 오고, 작성자는 "변호"가 아니라
  "설명"으로 응하면 건강한 리뷰가 됩니다.
</Paragraph>

<Paragraph>
  좋은 리뷰 코멘트는 보통 <strong>질문형</strong>이에요. "이 함수, `getUser`
  대신 `fetchUserById`가 더 명확할 것 같은데 어떠세요?" 같은 구체적인 제안. 나쁜
  리뷰는 "뭔가 이상한데요" 같은 막연한 불만. 주어가 <strong>작성한 사람</strong>
  이 아니라 <strong>"이 코드"</strong>에 찍히는 게 포인트입니다.
</Paragraph>

<Subtitle level={4}>리뷰에서 걸러지는 것들</Subtitle>

<Paragraph>
  실제로 리뷰에서 자주 잡히는 문제들을 짧게 리스트업해봅니다.
</Paragraph>

- <strong>네이밍 혼란</strong> — 앞의 2-3에서 본 `check(u, t)` 같은 것들
- <strong>중복 코드</strong> — 같은 로직이 세 군데에 복붙돼 있는 경우
- <strong>"이 코드 안전한가"</strong> — 사용자 입력을 검증 없이 받거나,
  비밀번호를 콘솔에 찍는 실수. 이 관점은 week9 보안 주차에서 본격적으로
  확장됩니다.
- <strong>테스트 누락</strong> — "이 기능 돌아가는 거 확인했나요?"

<Paragraph>
  특히 <strong>"안전한가"</strong> 관점은 주니어가 놓치기 쉬운 영역입니다. AI가
  써준 코드를 그대로 올렸을 때 가장 많이 지적받는 부분이기도 해요. week5
  대면에서 짧게 본 친구 게임 사례(점수 변수를 콘솔에서 백만으로 바꿔버린 얘기)가
  바로 이 지점입니다. week9 보안 대면에서 그 관점을 깊게 다룹니다.
</Paragraph>

<Subtitle level={4}>좋은 PR 설명이 리뷰 품질을 정한다</Subtitle>

<Paragraph>
  PR을 올릴 때 코드만 달랑 올리고 "확인 부탁드려요"라고 하면, 리뷰어는 수백 줄의
  코드를 맨땅에서 읽어야 합니다. 리뷰 품질이 떨어질 수밖에 없어요. 작성자가 먼저{" "}
  <strong>"뭘 왜 고쳤는지, 어디를 봐주면 좋은지"</strong>를 설명해주면, 리뷰어는
  그 맥락을 깔고 코드를 읽습니다. 같은 시간에 훨씬 깊은 리뷰가 나와요.
</Paragraph>

<Paragraph>
  실제 팀에서 쓰는 간단한 PR 설명 템플릿 예시입니다. 외울 필요는 없고, "이런
  틀이 있다" 정도만.
</Paragraph>

```yaml
## 무엇을
- 로그인 실패 시 에러 메시지를 콘솔이 아닌 UI에 표시

## 왜
- 사용자가 로그인 실패 사유를 알 수 없다는 제보 (#42)

## 리뷰 포인트
- 에러 메시지 문구가 적절한가
- 실패 시 입력 필드 포커스 이동이 UX에 맞는가
```

<Paragraph>
  실무 맥락으로 한 줄 덧붙이면, 협업 경험을 구체적으로 풀어내야 할 때{" "}
  <strong>
    "PR 리뷰를 이렇게 받았고, 이런 피드백으로 코드가 이렇게 바뀌었다"
  </strong>
  처럼 말할 수 있는 게 가장 설득력 있습니다. 리뷰 경험은 협업 능력의 증거처럼
  쓰입니다.
</Paragraph>

<Subtitle level={3}>
  2-6. 그라운드룰 샘플 — 실제 팀이 어떻게 운영되는가
</Subtitle>

<Paragraph>
  개념만 얘기하면 손에 안 잡히니, 실제 팀에서 쓰는{" "}
  <strong>그라운드룰 샘플</strong> 두 개만 보여드려요. 외우지 마세요. 팀마다
  다르고, 오늘 보는 건 "이런 모양의 규칙이 존재한다"는 예시입니다.
</Paragraph>

<Subtitle level={4}>
  샘플 ① 커밋 메시지 규칙 — Conventional Commits 맛보기
</Subtitle>

<Paragraph>
  커밋 제목 앞에 <strong>타입 접두사</strong>를 붙이는 방식입니다. 제목만 봐도
  "뭘 왜 바꿨는지"가 드러나는 게 목표예요. 외울 건 딱 두 개부터면 충분합니다.
</Paragraph>

- `feat:` — 새 기능 추가
- `fix:` — 버그 수정

<Paragraph>
  이 두 개만 알아도 팀플에서 당장 쓸 수 있어요. 문서 수정(`docs:`), 리팩터링
  (`refactor:`), 설정 변경(`chore:`) 같은 다른 접두사도 있지만, 필요할 때 그때
  찾아봐도 늦지 않습니다.
</Paragraph>

<Paragraph>실제 커밋 목록에 적히면 이런 모양이 됩니다.</Paragraph>

```bash
feat: 로그인 실패 에러 메시지 UI 표시
fix: 버튼 클릭 시 2번 submit되는 문제 해결
docs: README에 로컬 개발 세팅 추가
```

<Paragraph>
  이 규칙을 안 쓰면 커밋 목록이 "수정", "버그 고침", "ㅋㅋ 드디어" 같은 메시지로
  가득 차서, 반년 뒤에 "이 기능이 언제 들어왔지?"를 찾기가 어려워져요.
</Paragraph>

<Subtitle level={4}>샘플 ② 브랜치 네이밍 규칙</Subtitle>

<Paragraph>
  브랜치 이름에 <strong>"타입/주제"</strong> 구조를 씁니다. 하이픈으로 단어를
  잇고, 한국어는 툴 호환성 때문에 보통 피합니다.
</Paragraph>

- `feature/login-error-ui`
- `fix/double-submit`
- `refactor/user-api`

<Paragraph>
  이 이름만 봐도 "누가 뭘 하고 있는지" 팀원이 한 번에 알 수 있어요. 반대로
  브랜치 이름이 `test`, `test2`, `my-branch`처럼 지어지면 아무도 그 브랜치가 뭘
  하는지 모릅니다.
</Paragraph>

<Paragraph>
  핵심은 규칙 자체가 아니라 <strong>"팀원이 합의한 규칙이 있다"</strong>는
  사실입니다. 그라운드룰은 "이 팀이 어떻게 일하는지"를 정하는 문서예요. 실제
  팀에 들어가면 첫 PR에서 이 규칙을 안 지켜서 리뷰 멘트를 받는 경우가 꽤
  흔합니다. 미리 봐두면 그때 당황하지 않아요.
</Paragraph>

<Subtitle level={3}>2-7. AI와 팀 — AI가 쓴 코드를 PR에 올릴 때</Subtitle>

<Paragraph>
  week5 대면에서 <strong>"AI 시대에 왜 JS를 배우나"</strong> 얘기를 한 번
  꺼냈죠. 그 고리를 오늘 짧게 회수합니다. AI 보조 도구를 매일 쓰는 시대에,{" "}
  <strong>AI가 쓴 코드를 팀 PR에 올릴 때 뭘 조심해야 하는지</strong>.
</Paragraph>

<Subtitle level={4}>포인트 ① AI가 쓴 코드도 내 책임</Subtitle>

<Paragraph>
  PR을 올리는 순간 그 코드는 <strong>내 코드</strong>입니다.{" "}
  <strong>"AI가 써준 거라서 잘 모르겠어요"</strong>는 리뷰에서 통하지 않아요.
  이해한 코드만 올리는 게 기본 원칙입니다. AI가 10줄 써줬는데 내가 그중 3줄밖에
  설명 못 하겠다면, 3줄만 남기거나 전체를 이해한 뒤에 올리는 게 맞습니다.
</Paragraph>

<Paragraph>
  리뷰에서 "이 부분 왜 이렇게 짰어요?"라는 질문이 왔을 때{" "}
  <strong>"저도 잘 모르겠는데 AI가 이렇게 해줬어요"</strong>라고 답하는 순간, 팀
  입장에서는 그 코드가 <strong>리뷰 불가 상태</strong>가 됩니다.
</Paragraph>

<Subtitle level={4}>포인트 ② AI 표기는 팀마다 합의</Subtitle>

<Paragraph>
  AI 사용 여부를 PR이나 커밋에 표기할지는 팀마다 다릅니다. 일부 팀은 커밋
  메시지에 `Co-Authored-By: AI` 같은 표기를 요구하고, 일부는 "코드 품질만 맞으면
  상관없다"는 입장입니다. <strong>정답이 있는 주제가 아닙니다</strong>.
</Paragraph>

<Paragraph>
  단 업계 분위기로 몇 가지는 공통됩니다. 첫째,{" "}
  <strong>"AI 썼어요?"라고 먼저 밝히는 게 매너</strong>로 굳어지고 있어요.
  숨기는 게 문제가 되지, 밝히는 게 문제가 되는 경우는 드뭅니다. 둘째, 팀
  그라운드룰에 <strong>AI 사용 가이드</strong>가 포함되는 회사가 빠르게 늘고
  있습니다. 주니어로서의 기본 태도는 간단해요 —{" "}
  <strong>AI 썼다고 숨기지 말고 밝히되, 최종 코드의 책임은 본인이 진다</strong>.
</Paragraph>

<Subtitle level={4}>포인트 ③ AI가 기본적으로 안 챙기는 것</Subtitle>

<Paragraph>
  week5에서 본 친구 게임 사례(AI가 만든 웹 게임의 점수 함수가 전역에 노출돼
  콘솔에서 한 줄로 점수 폭주)를 다시 떠올려보세요. AI는{" "}
  <strong>"동작하는 코드"</strong>에 집중합니다.{" "}
  <strong>보안·엣지 케이스·팀 컨벤션</strong>은 기본적으로 먼저 챙기지 않아요.
  그래서 AI가 써준 코드를 그대로 PR에 올리면, 리뷰에서{" "}
  <strong>"입력 검증 빠졌어요", "우리 팀 네이밍 규칙이랑 다르네요"</strong> 같은
  멘트가 자주 붙습니다.
</Paragraph>

<Paragraph>
  즉 PR 리뷰는{" "}
  <strong>"AI가 기본적으로 안 챙기는 영역을 사람이 보충하는 자리"</strong>로
  기능합니다. 이게 앞으로 AI 시대에 주니어 개발자가 가질 수 있는 핵심 역할 중
  하나예요. "코드를 처음부터 치는 시간"은 줄고, "이 코드가 안전한지, 팀 스타일에
  맞는지 판단하는 시간"이 늘어납니다.
</Paragraph>

<Paragraph>
  논쟁 한 줄만 열어두고 갈게요 —{" "}
  <strong>"AI 표기가 필요한가"는 아직 의견이 갈리는 주제</strong>입니다. 정답이
  없고, 팀이 합의한 방식이 정답이에요. 오늘 대면에서 "나라면 어떻게 할까?"를
  서로 한마디씩 해봐도 좋습니다.
</Paragraph>

<Paragraph>
  이 "AI가 안 챙기는 영역" 감각은 <strong>week9 대면</strong>에서 본격적으로
  풀립니다. 오늘은 "PR 올릴 때 그 관점이 필요하다"는 한 줄만 예습으로
  남겨둡니다.
</Paragraph>

<Subtitle level={3}>week7에서 다루는 것</Subtitle>

- <strong>불변성</strong> — `const`가 왜 불변을 보장하지 않는지, 얕은 복사 vs
  깊은 복사, `Object.freeze`의 한계(얕은 동결)
- <strong>프로토타입</strong> — JS만의 상속 방식. 프로토타입 체인, 생성자 함수와
  `prototype`, `[[Prototype]]`와 `prototype`의 구분
- <strong>타입 체크</strong> — `typeof`의 한계(`typeof null === 'object'` 등),
  `Array.isArray`, `instanceof`, `Object.prototype.toString.call()`

<Paragraph>
  오늘 팀 컨벤션 얘기에서 <strong>`const`를 기본으로 쓰자</strong>는 관습을 잠깐
  언급했는데, week7은 그 `const`가 <strong>왜 완전 불변이 아닌지</strong>를 깊게
  파고듭니다. 오늘의 "주소가 복사된다"가 그대로 이어지는 주차예요.
</Paragraph>
