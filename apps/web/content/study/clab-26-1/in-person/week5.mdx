<Subtitle level={3}>핵심 리마인드</Subtitle>

<Subtitle level={4}>undefined vs null — 누가 준 값인가</Subtitle>

<Paragraph>
  <strong>undefined</strong>는 JS 엔진이 "아직 값을 안 넣었네"라고 자동으로
  넣어주는 값입니다. <strong>null</strong>은 개발자가 "여기는 일부러 비워놨어"
  라고 직접 넣는 값입니다. 비유하면 undefined는 택배가 아직 안 온 상태, null은
  "빈 상자를 일부러 보낸" 상태입니다.
</Paragraph>

```js
let a; // 선언만 하고 값을 안 넣음
console.log(a); // undefined — JS 엔진이 넣어줌

let b = null; // 개발자가 의도적으로 "비어있음"을 표시
console.log(b); // null
```

<Subtitle level={4}>var의 함수 스코프 — 블록을 무시하는 변수</Subtitle>

<Paragraph>
  `var`는 `if`·`for` 같은 블록을 스코프 경계로 인식하지 않습니다. 이게
  실질적으로 어떤 문제를 만드는지 코드로 보겠습니다.
</Paragraph>

```js
// var — if 블록을 무시하고 밖에서도 살아있음
if (true) {
  var secret = "비밀번호123";
}
console.log(secret); // "비밀번호123" — if 밖에서도 접근 가능

// let — 블록 안에서만 존재
if (true) {
  let safe = "안전한값";
}
console.log(safe); // ReferenceError — 블록 밖에서는 접근 불가
```

<Subtitle level={4}>호이스팅 시각화 — 엔진이 실제로 처리하는 순서</Subtitle>

<Paragraph>
  호이스팅은 "선언이 위로 끌어올려진다"라고 배우지만, 물리적으로 코드가 이동하는
  건 아닙니다. JS 엔진이 실행 전 <strong>생성 단계</strong>에서 스코프를 미리
  훑어 선언을 수집해두기 때문에 나타나는 현상입니다.
</Paragraph>

```js
// 우리가 작성한 코드
console.log(x); // undefined — 에러 아님!
var x = 5;
console.log(x); // 5

// JS 엔진이 이해하는 순서 (개념적)
// var x;              ← 생성 단계에서 선언 + undefined 초기화
// console.log(x);     ← undefined
// x = 5;              ← 실행 단계에서 할당
// console.log(x);     ← 5
```

<Subtitle level={4}>TDZ(Temporal Dead Zone) 시각화</Subtitle>

<Paragraph>
  여기가 이번 주 핵심입니다.{" "}
  <strong>같은 호이스팅인데 왜 var만 undefined가 되는가</strong>를 한 줄로
  정리하면: `var`는 선언과 `undefined` 초기화가 함께 호이스팅되고,
  `let`/`const`는{" "}
  <strong>
    선언만 호이스팅되고 초기화는 실제 선언문에 도달할 때 일어납니다
  </strong>
  . 이 "선언은 됐지만 초기화는 안 된" 구간이 TDZ 입니다.
</Paragraph>

```js
function example() {
  // ── 여기부터 TDZ 시작 (a는 선언은 됐지만 초기화 전)
  console.log(a); // ReferenceError — TDZ에서는 typeof도 에러
  let a = 1;
  // ── 여기부터 TDZ 끝 (a 초기화 완료)
  console.log(a); // 1
}
```

<Paragraph>
  TDZ 덕분에 `let`/`const`는 예상치 못한 `undefined`가 조용히 코드에 흘러들어
  버그가 되는 일이 원천 차단됩니다. 에러를 <strong>지금, 여기서</strong> 터뜨려
  주는 것이 더 안전하다는 설계입니다.
</Paragraph>

<Subtitle level={4}>NaN === NaN이 false인 이유</Subtitle>

<Paragraph>
  <strong>NaN(Not a Number)</strong>은 "계산 실패"를 뜻합니다. 실패한 계산 두
  개가 같은 실패인지 비교하는 건 의미가 없으므로, IEEE 754 표준에서 NaN은 자기
  자신과도 같지 않다고 정의했습니다. NaN 여부 확인에는 반드시{" "}
  <strong>`Number.isNaN()`</strong>을 씁니다.
</Paragraph>

```js
NaN === NaN; // false — 자기 자신과도 같지 않은 유일한 값
Number.isNaN(NaN); // true — 올바른 확인 방법
isNaN("hello"); // true — 전역 isNaN은 타입 변환을 해버려서 위험!
```

<Subtitle level={3}>2-1. 왜 이 얘기를 지금 하는가</Subtitle>

<Paragraph>
  "AI가 코딩을 다 한다"는 말이 과장이라고 생각했던 제가, 지금은 AI를 안쓰면

  일어나고 있어서, JS를 배우는 것 자체, 개발을 배우는 것 자체에 대한 회의감이 들

  배울지 얘기해보려고 합니다.
</Paragraph>

<Subtitle level={3}>
  2-2. 웹 개발 프로세스 5단계 — 프론트가 어디에 있는가
</Subtitle>

<Paragraph>
  웹 서비스 하나가 세상에 나오기까지, 현업에서는 대략 다섯 단계를 거칩니다.
  회사마다 경계가 흐릿해지기도 하지만 큰 그림은 비슷합니다.
</Paragraph>

- <strong>기획 (PM/PO)</strong> — "무엇을 왜 만드는가"를 정의. 요구사항 문서,
  와이어프레임이 여기서 나옵니다.
- <strong>디자인 (디자이너)</strong> — Figma 같은 도구로 UI 시안을 제작. 색상,
  여백, 컴포넌트 규격이 정해집니다.
- <strong>프론트엔드 (우리)</strong> — 시안을 HTML/CSS/JS로 구현하고, 백엔드와
  데이터를 주고받아 화면을 살아있게 만듭니다.
- <strong>백엔드</strong> — 데이터 저장·조회 로직과 API를 만듭니다. DB가 여기에
  붙습니다.
- <strong>배포·운영 (DevOps)</strong> — 만든 걸 서버에 올리고, 에러를 모니터링
  하고, 성능을 관리합니다.

<Paragraph>
  이 중 우리가 하는 일을 한 문단으로 풀면 이렇습니다.{" "}
  <strong>
    디자인 시안을 받아서 HTML/CSS로 화면을 그리고, JS로 동작을 붙이고, 백엔드
    API로 데이터를 주고받은 뒤 배포
  </strong>
  한다. 이 네 가지(그리기 · 동작 · 통신 · 배포)가 프론트의 뼈대입니다.
</Paragraph>

<Paragraph>
  앞으로 이 네 가지가 각각 어느 주차에 등장하는지만 예고하자면,{" "}
  <strong>"동작 붙이기"는 이번 주 시작한 JS가 계속 이어지고</strong>, "데이터
  주고받기"는 통신/API 주차에, "배포"는 스터디 후반부의 Vercel·Docker 주차에서
  다시 만납니다. 각 주차가 오면 "아, 그때 그 얘기구나" 하는 감각이 조금씩 쌓일
  거예요.
</Paragraph>

<Subtitle level={3}>2-3. 프론트엔드 스택의 역사 — 왜 React가 나왔나</Subtitle>

<Paragraph>
  프론트엔드 기술은 10여 년 사이에 스타일이 크게 세 번 바뀌었습니다. 각 단계가
  <strong>"이전 방식의 어떤 불편함"</strong>을 해결하려다 나왔는지 맥락만
  짚겠습니다. 기술 이름은 지금 외우지 않아도 됩니다.
</Paragraph>

- <strong>바닐라 JS + jQuery 시대 (~2010년대 초)</strong> — "버튼 클릭하면 이
  DOM을 직접 찾아서 바꿔라"를 손으로 썼습니다. 화면이 복잡해질수록 유지보수가
  빠르게 지옥이 됩니다.
- <strong>React/Vue 등장 (2013~)</strong> — "상태가 바뀌면 화면이 알아서 다시
  그려진다"는 발상. 직접 DOM을 조작하는 게 아니라 "지금 상태가 이러니까 화면은
  이래야 해"를 선언만 합니다. 이를 <strong>선언형</strong>이라고 부릅니다.
- <strong>Next.js / Vite 시대 (2020~)</strong> — React로 앱을 그냥 만들면 느리고
  복잡합니다. 서버에서 미리 렌더링하거나(SSR), 빠르게 번들링하는 일을 기본으로
  해주는 <strong>메타 프레임워크</strong>·<strong>번들러</strong>가 기본값이
  됐습니다.

<Paragraph>
  한 줄 대비만 해보면 이런 느낌입니다. 옛날 방식은 우리가 이미 DOM을 알고 있으니
  코드로도 보여줄게요.
</Paragraph>

```js
// 옛날식: 내가 DOM을 직접 찾아서 바꿈
document.getElementById("count").innerText = count + 1;
```

<Paragraph>
  반면 React 방식은 한 문장으로 요약됩니다.{" "}
  <strong>
    "상태(count)의 값만 바꿔라. 그러면 화면은 알아서 다시 그려진다"
  </strong>
  . 개발자가 직접 DOM을 찾아 바꾸는 줄이 사라집니다. 아직 React 문법을 안
  배웠으니 코드는 보여드리지 않습니다 — 지금은 "DOM을 손으로 바꾸던 일을
  프레임워크가 대신해준다"는 문장만 가져가세요.
</Paragraph>

<Paragraph>
  지금은 두 번째 스타일이 "이상하다" 싶을 수 있는데요, week10쯤 React를 본격
  소개할 때 "아, 그때 그 '알아서 다시 그려진다'는 얘기!" 하는 감각이 돌아오면
  충분합니다.
</Paragraph>

<Subtitle level={3}>
  2-4. AI 활용 근황 — 2026년 개발자는 어떻게 일하는가
</Subtitle>

<Paragraph>
  "지금 JS를 공부하면 AI 시대에도 의미가 있나요?" 라는 질문을 자주 받습니다.
  오늘 이 주제를 여기서 꺼내는 이유는,{" "}
  <strong>이 질문에 답을 해두지 않으면 앞으로 8주가 피로해지기 때문</strong>
  입니다. 뉴스에 매일 "AI가 코딩 다 한다"가 나오는데, 그 와중에 우리는{" "}
  <code>var</code>와 <code>let</code> 차이를 배우고 있을 테니까요.
</Paragraph>

<Subtitle level={4}>
  실제 있었던 일 — 친구가 만든 웹 게임을 5분 만에 깬 이야기
</Subtitle>

<Paragraph>
  본론 들어가기 전에 짧은 에피소드 하나부터 볼게요. 웹 개발을 모르는 친구가 AI로
  웹 게임을 만들어 왔습니다. 점수 겨루기 게임이었어요. 그런데 브라우저 F12로
  열어보니 <strong>소스 코드가 다 보였습니다</strong>. 랭킹을 올리려고 뭘
  했냐면, 그냥 브라우저 콘솔에서 점수 변수를 백만으로 바꿔버렸어요. 5분 만에
  1등.
</Paragraph>

<Paragraph>
  "에이 그건 옛날 얘기 아냐?"가 아닙니다.{" "}
  <strong>
    AI한테 만들어달라고 한 코드는 기본적으로 이런 허점을 막아주지 않습니다
  </strong>
  . 왜냐하면 AI는 "게임이 작동하는 코드"를 만들어주는 데 집중하지, "나쁜 사람이
  이걸 어떻게 악용할까"는 먼저 생각하지 않거든요. 한 단계 더 나가볼게요. F12
  콘솔에서 이렇게 쳐봤더니 이런 일이 벌어졌습니다.
</Paragraph>

<Paragraph>
  아래 코드는 <strong>지금 전부 이해하지 않아도 됩니다</strong>. "게임 안의 함수
  이름을 그대로 콘솔에서 불렀다"는 것만 눈치채면 충분해요.
</Paragraph>

```js
// 게임이 내부에서 쓰는 함수가 그대로 전역에 노출돼 있었다
applyLaser(anyRegion, anyRegion); // 1회 호출에 점수 +38

// 같은 호출을 반복하면 점수가 계속 올라가는 구조였습니다
```

<Paragraph>
  게임이 내부 로직을 위해 쓰는 <code>applyLaser</code> 같은 함수가{" "}
  <strong>전역(window)에 그대로 붙어 있었고</strong>,{" "}
  <strong>"레이저 특수 아이템을 진짜로 썼을 때만 발동해야 한다"는 검증</strong>
  이 한 줄도 없는 상태였습니다. 아무 셀 객체를 넣어도 점수가 올라갔어요. 만든
  친구는 이런 경로가 있다는 것조차 몰랐고, AI도 먼저 알려주지 않았던 거죠.
</Paragraph>

<Paragraph>
  이게 오늘 이야기의 출발점입니다. AI는 동작하는 코드를 빠르게 만들어주는데요,{" "}
  <strong>무엇이 위험한지까지 먼저 챙겨주지는 않습니다</strong>. 그래서 우리가
  알아야 해요. 지금부터 그 얘기를 하나씩 풀어갈게요.
</Paragraph>

<Subtitle level={4}>(가) 실무 개발자는 AI를 어떻게 쓰고 있는가</Subtitle>

<Paragraph>
  한 줄로 요약하면, <strong>요즘은 CLI가 거의 다 먹었다</strong>는 흐름입니다.
  2~3년 전만 해도 에디터 안에서 자동완성을 받는 방식(github copilot 같은 확장)이
  주류였는데요, 지금은 터미널에서 AI 에이전트가 파일을 직접 읽고 수정하는{" "}
  <strong>CLI 도구</strong>로 무게중심이 빠르게 이동했습니다.
</Paragraph>

<Paragraph>
  대표적인 CLI 도구가 <strong>claude code, codex cli, gemini cli</strong> 같은
  친구들입니다. 터미널에 명령 한 줄 쓰면, 에이전트가 프로젝트를 훑어보고 원하는
  파일을 찾아 수정한 뒤 결과를 보고해요. "내가 코드를 붙여넣고 답을 받는다"가
  아니라, <strong>"에이전트가 내 코드베이스에서 일한다"</strong>는 쪽에
  가깝습니다. 에디터 기반 도구(github copilot, cursor)도 여전히 쓰이지만, 비중은
  점점 CLI로 넘어가는 중이에요.
</Paragraph>

<Paragraph>
  이 변화가 왜 생겼는지 한 줄만 덧붙이면 — CLI는{" "}
  <strong>프로젝트 전체를 맥락</strong>
  으로 다루기가 훨씬 쉽고, 스크립트나 CI와도 자연스럽게 엮입니다. 그래서 요즘
  팀들은 "얘한테 어디서부터 어디까지 맡길까"를 설계하는 데 시간을 쓰죠.
</Paragraph>

<Paragraph>
  공통점은 도구가 바뀌어도 한 가지예요.{" "}
  <strong>
    "코드를 처음부터 손으로 치는 시간"은 줄고, "이 코드가 왜 이렇게 생겼는지
    판단하는 시간"이 늘었습니다
  </strong>
  . 역설적이지만 그래서 JS 문법 감각이 더 중요해졌어요. AI가 써준 코드를 읽고
  "이건 좀 이상한데"를 잡아내는 게 주니어의 실력을 가르는 지점이 되고 있습니다.
</Paragraph>

<Paragraph>
  참고로 방금 언급한 CLI 도구들(claude code, codex cli)이 내부에서 쓰는 개념이
  바로 아래 (나)에서 볼 <strong>에이전트(agent)</strong>입니다. 도구 이름과 개념
  이름이 겹쳐 보일 수 있는데,{" "}
  <strong>도구가 여러 에이전트를 묶어 쓰는 구조</strong>라고 보면 정리가 돼요.
</Paragraph>

<Subtitle level={4}>(나) 에이전트와 하네스 — 한 명의 AI가 아니라, 팀</Subtitle>

<Paragraph>
  조금 더 최신 얘기입니다. 요즘은 AI 하나에게 "다 해줘" 대신,{" "}
  <strong>여러 AI를 각자 다른 역할로 엮어서 협업</strong>시키는 패턴이 떠오르고
  있습니다. 이걸 <strong>에이전트(agent)</strong>와{" "}
  <strong>하네스(harness)</strong>라고 부릅니다. 말이 낯설 뿐, 비유는 간단해요.
</Paragraph>

- <strong>에이전트</strong> = 역할이 정해진 AI 한 명 (예: "코드 작성자",
  "리뷰어", "테스트 러너")
- <strong>하네스</strong> = 여러 에이전트를 엮어주는 구조. 순서·검증·재시도를
  관리.

<Paragraph>
  포인트는 이겁니다.{" "}
  <strong>
    AI 한 명에게 "이 기능 다 만들어줘" 하면 한 번에 쓰는 만큼 허점도 많이
    섞입니다
  </strong>
  . 근데 "A 에이전트는 기능 목록을 만들고 → B 에이전트는 그 기능을 코드로 쓰고 →
  C 에이전트는 그 코드를 테스트한다"로 <strong>역할을 쪼개면</strong> 훨씬
  안정적인 결과가 나옵니다. 이게 하네스의 기본 감각이에요. 주니어 입장에서는
  지금 당장 만들 필요는 없고,{" "}
  <strong>"AI를 한 명이 아니라 팀으로 쓰는 접근이 있다"</strong> 정도만 알아두면
  됩니다.
</Paragraph>

<Subtitle level={4}>(다) "AI 썼어요"를 숨기느냐, 드러내느냐</Subtitle>

<Paragraph>
  불과 1~2년 전만 해도 분위기가 달랐어요. 개인 프로젝트 GitHub에 올릴 때 AI 설정
  파일(<code>.claude</code>, <code>CLAUDE.md</code> 같은 것들)을{" "}
  <strong>.gitignore에 넣어서 숨기는</strong> 게 흔했습니다. "AI 썼다는 티를
  내면 실력이 없어 보이지 않을까"하는 분위기였죠. 저도 그랬어요.
</Paragraph>

<Paragraph>
  지금은 반대로 가고 있습니다. 설정 파일을 <strong>그대로 커밋</strong>하고,
  README에 "이 프로젝트에서 AI를 이렇게 썼다"를 적는 쪽이 늘고 있어요. 흐름을 한
  줄로 정리하면{" "}
  <strong>"AI를 썼느냐"가 아니라 "어떻게 썼느냐"로 관심이 옮겨갔다</strong>는
  겁니다. 이유도 단순해요. AI를 안 쓰는 개발자가 드물어지니까, "썼다"는 사실
  자체는 정보가 아니게 된 거죠.
</Paragraph>

<Paragraph>
  그럼 뭐가 정보가 됐냐면, <strong>쓰는 방식</strong>입니다. 어떤 역할을
  맡겼는지, 어디를 믿고 어디를 직접 검토했는지, 어떤 실수가 나올 줄 알고 어떻게
  걸러냈는지. 채용 쪽 얘기를 들어봐도 "AI 써봤어요"보다{" "}
  <strong>"이 부분은 AI가 놓칠 걸 알고 따로 확인했어요"</strong>라고 말할 수
  있는 쪽을 높게 봅니다. 아까 (나)에서 본 "한 명 대신 팀으로 쓰기"도 같은
  맥락이에요.
</Paragraph>

<Paragraph>
  단, <strong>공개할 때 같이 챙겨야 할 게 하나</strong> 있어요.{" "}
  <strong>
    설정 파일에 내부 API 주소·비밀 엔드포인트·팀 내부 규칙이 적혀 있으면 안 된다
  </strong>
  는 점입니다. 공개 레포에서는 이게 훨씬 더 쉽게 새나가는 경로가 되거든요.
  커밋하기 전에 한 번씩 훑는 습관만 붙여두면 됩니다.
</Paragraph>

<Paragraph>
  정리하면, 스터디 과제를 올릴 때도 AI를 숨길 필요는 없습니다. 오히려{" "}
  <strong>"어디서 어떻게 AI를 썼다"를 짧게 적어두면 그게 기록</strong>이 돼요.
  나중에 포트폴리오로 보여줄 때도 그 기록이 본인이 어떻게 일하는지를 대신
  말해줍니다.
</Paragraph>

<Subtitle level={4}>(라) "그럼 왜 JS를 배우나요"에 답</Subtitle>

<Paragraph>세 가지입니다.</Paragraph>

1. <strong>AI가 쓴 코드를 읽을 줄 알아야 한다.</strong> 문법 감각이 없으면 AI가
   써준 코드가 맞는지 틀린지 모릅니다. 그러면 AI를 쓸 수 없어요. "프롬프트만 잘
   쓰면 된다"는 말은 <strong>읽을 줄 아는 사람만 해당</strong>됩니다.
2. <strong>AI가 기본적으로 막아주지 않는 영역이 있다.</strong> 대표적인 게
   <strong>보안</strong>입니다. 맨 앞에서 본 친구 게임처럼, XSS·CSRF·토큰 보관
   실수는 AI가 먼저 알려주지 않습니다. 이건 <strong>week9 대면</strong>에서
   배웁니다. (week9 비대면은 JS의 클래스 문법을 다루고, 대면에서는 보안을 다루는
   두 갈래 구조예요. 지금은 "AI도 모르는 영역이 있구나" 정도만 기억.)
3. <strong>신입한테 요구하는 수준이 계속 올라가고 있다.</strong> 실무의 속도가
   빨라졌다는 건 사실이에요. 그런데 "이제 기초 3개월이면 취업"이냐고 하면 그건
   아닙니다. 오히려 반대예요. AI로 빠르게 만드는 게 당연해지면서, 예전에는 경력
   1~2년차에게 기대하던 것들(협업, 코드 리뷰, 보안 감각, 성능 측정)이
   <strong>신입한테도 자연스럽게 요구</strong>되고 있어요. 기초가 탄탄하지
   않으면 "AI로 화면은 만들었는데 리뷰에서 막히는" 상태에 쉽게 갇힙니다. 그래서
   이번 8주를 촘촘하게 가져가는 거예요.

<Paragraph>
  정리하면 <strong>"AI를 쓰되, 쓰기 위해 기초를 튼튼히 한다"</strong>입니다.
  주차 과제를 풀 때 AI를 쓰는 건 자유롭게 허용합니다. 단, 스킵 진단 10문제와
  학습 이해 퀴즈는 <strong>꼭 본인이 먼저</strong> 풀어보세요. 거기서 막히는
  포인트가 진짜 배워야 할 지점이에요.
</Paragraph>
