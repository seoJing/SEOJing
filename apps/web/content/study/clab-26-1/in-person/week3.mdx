<Subtitle level={2}>1. 저번 주 복습</Subtitle>

<Paragraph>
  지난주에는 사용자로부터 정보를 입력받는 <strong>폼(Form)</strong>과, CSS를
  HTML에 연결하고 원하는 요소를 정확히 선택하는 <strong>CSS 선택자</strong>를
  배웠습니다. 핵심만 빠르게 짚어보겠습니다.
</Paragraph>

<Subtitle level={3}>폼(Form) 핵심</Subtitle>

- <strong>`form` 태그</strong>: 입력 요소들을 감싸는 컨테이너입니다.
  `action`으로 전송할 URL을, `method`로 전송 방식(GET/POST)을 지정합니다.
- <strong>`input` 태그</strong>: `type` 속성으로 텍스트, 비밀번호, 이메일,
  체크박스, 라디오 등 다양한 입력 방식을 결정합니다.
- <strong>`name` 속성</strong>: 서버로 데이터를 전송할 때 "키(key)" 역할을
  합니다. `name`이 없으면 해당 입력값은 서버로 전송되지 않습니다.
- <strong>`label`과 `for`</strong>: `label`의 `for` 값과 `input`의 `id` 값을
  일치시키면 클릭 시 해당 입력 요소에 포커스가 이동합니다.
- <strong>`select`, `textarea`</strong>: 드롭다운 선택과 여러 줄 텍스트 입력을
  담당합니다.

<Subtitle level={3}>CSS 선택자 핵심</Subtitle>

- <strong>태그 선택자</strong>: `p {}` — 해당 태그 전부 선택
- <strong>클래스 선택자</strong>: `.highlight {}` — 마침표(.)로 시작
- <strong>아이디 선택자</strong>: `#main-title {}` — 샵(#)으로 시작
- <strong>자손 결합자</strong>: `nav a {}` — 내부 모든 요소 선택
- <strong>자식 결합자</strong>: `ul > li { }` — 직계 자식만 선택
- <strong>우선순위</strong>: 인라인 스타일 > 아이디(#) > 클래스(.) > 태그

<Subtitle level={3}>학습 자료에서 다룬 추가 개념</Subtitle>

<Paragraph>
  2주차 학습 자료에서 강의 외 추가 개념들을 다뤘습니다. 다시 한번
  짚어보겠습니다.
</Paragraph>

- <strong>GET vs POST</strong>: 둘 다 데이터를 서버로 보내지만, GET은 URL
  쿼리 스트링(`?q=검색어`)에 붙어 노출되고, POST는 요청 본문에 숨겨집니다.
  "주소창에 보여도 괜찮으면 GET, 아니면 POST"가 판단 기준입니다. GET은
  URL 공유·북마크·캐싱이 가능하다는 고유한 장점이 있어서 검색·필터에
  적합합니다.
- <strong>`fieldset`과 `legend`</strong>: 회원가입처럼 입력 필드가 많을 때
  "기본 정보", "보안 설정"처럼 관련 항목을 그룹화합니다. 스크린 리더가
  `legend`를 먼저 읽어주기 때문에 접근성에도 도움이 됩니다.
- <strong>CSS 적용 세 가지 방법</strong>: 인라인 스타일(`style=""`), 내부 스타일
  시트(`<style>`), 외부 스타일 시트(`<link>`) 중 외부 스타일 시트가 유지보수와
  재사용성 면에서 가장 권장됩니다. 인라인 스타일은 우선순위가 너무 높아
  디버깅이 어려워지므로 가급적 피합니다.
- <strong>CSS 선택자 우선순위</strong>: 인라인 스타일 > 아이디(#) > 클래스(.) >
  태그 순서입니다. "CSS를 썼는데 왜 안 먹히지?" 싶으면 우선순위 충돌을
  의심해보세요.

<Subtitle level={2}>2. 저번 주 과제 정답 및 풀이</Subtitle>

<Subtitle level={3}>학습 자료 2주차 과제</Subtitle>

<Paragraph>
  <strong>
    Q. 회원가입 폼에서 `&lt;input type="text"&gt;`로 비밀번호 입력칸을 만들면
    기능적으로 동작은 하지만, 보안과 사용자 경험 측면에서 어떤 문제가 생길까요?
    또한, `type="email"`을 사용하면 단순히 `type="text"`를 쓸 때와 비교해
    브라우저가 어떤 추가적인 도움을 줄까요?
  </strong>
</Paragraph>

<Subtitle level={4}>`type="text"`로 비밀번호를 받으면?</Subtitle>

- <strong>화면 노출</strong>: 입력한 비밀번호가 그대로 화면에 보입니다. 주변
  사람이나 화면 녹화 등으로 비밀번호가 노출될 수 있습니다.
- <strong>자동완성 문제</strong>: 브라우저의 비밀번호 관리자가 이 필드를
  비밀번호로 인식하지 못합니다. "비밀번호를 저장하시겠습니까?" 팝업이 뜨지 않고,
  자동 입력도 되지 않습니다.
- <strong>접근성</strong>: 스크린 리더가 이 필드를 일반 텍스트 입력으로
  안내합니다. 사용자가 민감한 정보를 입력하고 있다는 사실을 인지하기 어렵습니다.

<Paragraph>
  `type="password"`를 사용하면 입력값이 마스킹(●)되고, 브라우저가 비밀번호
  필드로 인식하여 자동완성과 보안 기능을 제공합니다.
</Paragraph>

<Subtitle level={4}>`type="email"`의 추가 도움</Subtitle>

- <strong>자동 유효성 검사</strong>: 브라우저가 `@`가 포함되었는지 등 기본적인
  이메일 형식을 자동으로 검증합니다. JavaScript를 쓰지 않아도 됩니다.
- <strong>모바일 키보드 최적화</strong>: 스마트폰에서 `@`와 `.com` 버튼이 포함된
  이메일 전용 키보드가 나타납니다.
- <strong>자동완성</strong>: 브라우저가 이전에 입력한 이메일 주소를
  제안해줍니다.

<Paragraph>
  핵심은 <strong>올바른 `type`을 지정하면 브라우저가 무료로 도와준다</strong>는
  것입니다. 코드 한 단어 차이로 보안, 유효성 검사, 사용자 경험이 모두
  개선됩니다.
</Paragraph>

<Subtitle level={2}>3. 면접 질문</Subtitle>

<Paragraph>
  이번 주 범위와 관련된 실제 면접 질문들을 함께 생각해봅시다. 정답을
  외우기보다는 자신만의 말로 설명하는 연습을 해보세요.
</Paragraph>

<Subtitle level={3}>Q1. GET과 POST의 차이를 설명해주세요.</Subtitle>

<Paragraph>
  둘 다 HTTP 요청 방식이지만, <strong>데이터를 실어 보내는 위치</strong>가
  다릅니다.
</Paragraph>

- <strong>GET</strong>: 데이터가 URL의 쿼리 스트링에 붙어서 전송됩니다
  (`?key=value`). URL에 노출되므로 민감하지 않은 데이터(검색, 필터링)에
  적합합니다. URL 공유, 북마크, 브라우저 캐싱이 가능한 것이 장점입니다.
- <strong>POST</strong>: 데이터가 HTTP 요청 본문(Body)에 담겨 전송됩니다. URL에
  노출되지 않아 로그인, 회원가입, 결제 등 민감한 정보에 적합합니다.

<Paragraph>
  면접 팁: "주소창에 보여도 괜찮으면 GET, 아니면 POST"라고 정리하면 간결합니다.
  다만 POST도 HTTPS 없이는 안전하지 않다는 점도 언급하면 좋습니다.
</Paragraph>

<Subtitle level={3}>
  Q2. CSS 선택자 우선순위(Specificity)를 설명해주세요.
</Subtitle>

<Paragraph>
  같은 요소에 여러 CSS 규칙이 적용될 때, 어떤 스타일이 최종 적용될지 결정하는
  규칙입니다.
</Paragraph>

- <strong>우선순위 순서</strong>: `!important` > 인라인 스타일 > 아이디(#) >
클래스(.) / 속성 / 가상 클래스 > 태그 / 가상 요소
- 같은 우선순위라면 <strong>나중에 작성된 스타일</strong>이 적용됩니다.
- `!important`는 우선순위를 강제로 끌어올리지만, 디버깅이 극도로 어려워지므로
  실무에서는 가급적 사용하지 않습니다.

<Subtitle level={3}>Q3. 박스 모델(Box Model)이 무엇인가요?</Subtitle>

<Paragraph>
  모든 HTML 요소는 사각형 상자로 이루어져 있습니다. 이 상자는 안쪽부터
  바깥쪽으로 네 영역으로 구성됩니다.
</Paragraph>

1. <strong>Content</strong>: 텍스트, 이미지 등 실제 내용이 들어가는 영역
2. <strong>Padding</strong>: 콘텐츠와 테두리 사이의 안쪽 여백 (배경색 적용됨)
3. <strong>Border</strong>: 요소의 테두리
4. <strong>Margin</strong>: 테두리 바깥쪽의 다른 요소와의 간격 (투명)

<Paragraph>
  `box-sizing: content-box`(기본값)에서는 width가 콘텐츠 영역만을 의미하고,
  `box-sizing: border-box`에서는 padding과 border까지 포함합니다. 현대 웹
  개발에서는 `border-box`를 기본으로 설정하는 것이 표준입니다.
</Paragraph>

<Subtitle level={2}>4. 보충: script 로딩 전략 (defer, async)</Subtitle>

<Paragraph>
  2주차 학습 자료에서 `defer`를 잠깐 언급했는데, 이 개념을 조금 더 깊이
  파보겠습니다. 브라우저가 HTML을 위에서 아래로 읽다가 `<script>` 태그를 만나면
  어떤 일이 일어나는지가 핵심입니다.
</Paragraph>

<Subtitle level={3}>세 가지 로딩 방식</Subtitle>

- <strong>`<script>`(기본)</strong>: HTML 파싱을 <strong>멈추고</strong>,
스크립트를 다운로드한 뒤 실행합니다. 실행이 끝나야 나머지 HTML을 파싱합니다.
그래서 `<body>` 끝에 두는 관행이 생겼습니다.
- <strong>`<script defer>`</strong>: 스크립트를 <strong>백그라운드에서
다운로드</strong>하면서 HTML 파싱을 계속합니다. HTML 파싱이 끝난 뒤에 스크립트를
실행합니다. 여러 개의 defer 스크립트는 <strong>작성 순서대로</strong>
실행됩니다.
- <strong>`<script async>`</strong>: 마찬가지로 백그라운드에서 다운로드하지만,
다운로드가 끝나면 <strong>즉시 실행</strong>합니다. 실행 순서가 보장되지
않으므로, 다른 스크립트에 의존하지 않는 독립적인 스크립트(광고, 분석 도구 등)에
적합합니다.

```html
<!-- 기본: HTML 파싱을 막음 -->
<script src="app.js"></script>

<!-- defer: HTML 파싱 완료 후 순서대로 실행 -->
<script defer src="app.js"></script>

<!-- async: 다운로드 완료 즉시 실행, 순서 보장 안 됨 -->
<script async src="analytics.js"></script>
```

<Subtitle level={3}>React / Next.js에서는?</Subtitle>

<Paragraph>
  React 프로젝트에서는 번들러(Vite, Webpack 등)가 빌드한 JS 파일을 자동으로
  적절한 위치에 삽입해주기 때문에, 직접 defer/async를 걱정할 일이 거의 없습니다.
</Paragraph>

<Paragraph>
  하지만 외부 스크립트(Google Analytics, 카카오 SDK 등)를 직접 넣어야 할 때는
  이야기가 달라집니다. Next.js는 이를 위해 `next/script` 컴포넌트를 제공합니다.
</Paragraph>

```jsx
import Script from "next/script";

// defer처럼 동작: HTML 파싱 후 실행
<Script src="https://example.com/sdk.js" strategy="afterInteractive" />

// async처럼 동작: 브라우저가 한가할 때 실행
<Script src="https://example.com/analytics.js" strategy="lazyOnload" />
```

<Paragraph>
  원리를 알아야 "외부 스크립트를 넣었더니 페이지가 느려졌다"는 상황에서 올바른
  전략을 선택할 수 있습니다.
</Paragraph>

<Subtitle level={2}>6. 다음 주 안내</Subtitle>

<Paragraph>
  다음 주에는 CSS의 꽃인 <strong>레이아웃</strong>을 본격적으로 다룹니다.
  Position으로 요소를 원하는 위치에 배치하고, <strong>Flexbox</strong>로 유연한
  가로·세로 레이아웃을 구성하는 것이 목표입니다.
</Paragraph>

<Subtitle level={3}>다음 주 영상 예습</Subtitle>

<Paragraph>
  다음 주 영상(4주차 진도: 04:28:10 ~ 05:28:02)에서는 CSS의 꽃인 레이아웃에
  들어갑니다. 영상을 보기 전에 아래 내용을 가볍게 훑어두면 이해가 빠릅니다.
</Paragraph>

- <strong>배경 이미지</strong>: `background-image`로 요소에 이미지를 넣고,
  `background-size`, `background-repeat` 등으로 크기와 반복을 제어하는 법을
  배웁니다.
- <strong>Position</strong>: 요소를 문서 흐름에서 꺼내서 원하는 위치에 배치하는
  속성입니다. `static`, `relative`, `absolute`, `fixed` 네 가지가 나오는데,
  각각 <strong>"기준점이 어디인가"</strong>가 다릅니다.
- <strong>Flexbox</strong>: 이번 주 가장 중요한 개념입니다. 자식 요소들을
  가로 또는 세로로 유연하게 정렬하는 레이아웃 시스템입니다. 내비게이션 바,
  카드 배치, 정중앙 정렬 등 거의 모든 레이아웃에 쓰입니다.
