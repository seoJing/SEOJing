<Subtitle level={2}>1. 저번 주 복습</Subtitle>

<Paragraph>
  지난주에는 사용자로부터 정보를 입력받는 <strong>폼(Form)</strong>과, CSS를
  HTML에 연결하고 원하는 요소를 정확히 선택하는 <strong>CSS 선택자</strong>를
  배웠습니다. 이번에는 단순 나열이 아니라, 각 개념이{" "}
  <strong>왜 중요한지</strong>와 <strong>실수하기 쉬운 포인트</strong>를
  중심으로 복습하겠습니다.
</Paragraph>

<Subtitle level={3}>폼(Form) 핵심</Subtitle>

<Subtitle level={4}>`form` 태그와 `action`, `method`</Subtitle>

<Paragraph>
  `form` 태그는 입력 요소들을 감싸는 컨테이너입니다. <strong>택배 상자</strong>
  에 비유하면, `action`은 <strong>배송지 주소</strong>이고 `method`는
  <strong>배송 방법</strong>(일반 택배 vs 등기)입니다. 상자 없이 물건만 던지면
  배달이 안 되듯, `form` 없이 `input`만 있으면 데이터가 서버로 전송되지
  않습니다.
</Paragraph>

```html
<form action="/signup" method="post">
  <input type="text" name="username" />
  <input type="password" name="password" />
  <button type="submit">회원가입</button>
</form>
```

<Subtitle level={4}>`name` 속성 — 없으면 서버로 안 간다</Subtitle>

<Paragraph>
  `name` 속성은 서버로 데이터를 보낼 때 <strong>키(key)</strong> 역할을 합니다.
  택배 송장에 <strong>받는 사람 이름이 없는 것</strong>과 같습니다. 택배가
  도착해도 누구 것인지 모르면 전달할 수 없듯, `name`이 없는 `input`의 값은
  서버에 전혀 전달되지 않습니다.
</Paragraph>

```html
<!-- name이 없어서 서버로 전송 안 됨 -->
<input type="text" />

<!-- name이 있어서 서버에 username=입력값 형태로 전송 -->
<input type="text" name="username" />
```

<Paragraph>
  <strong>흔한 실수</strong>: 폼을 만들고 "서버에 데이터가 안 들어와요"라고 할
  때, 가장 먼저 확인할 것이 `name` 속성 누락입니다. 개발자 도구의 Network 탭에서
  Form Data를 보면 바로 확인할 수 있습니다.
</Paragraph>

<Subtitle level={4}>`label`과 `for` — 클릭 면적을 넓혀주는 연결</Subtitle>

<Paragraph>
  `label`의 `for` 값과 `input`의 `id` 값을 일치시키면,{" "}
  <strong>label 텍스트를 클릭해도 input에 포커스</strong>가 이동합니다.
  데스크톱에서는 편의 기능이지만, 모바일에서는{" "}
  <strong>터치 영역이 넓어지는 실질적인 UX 개선</strong>입니다. 작은 체크박스나
  라디오 버튼을 손가락으로 정확히 누르기 어려운 상황을 떠올려보세요.
</Paragraph>

```html
<!-- label을 클릭하면 id="email" input에 포커스 -->
<label for="email">이메일 주소</label>
<input type="email" id="email" name="email" />

<!-- 체크박스: label 클릭으로 체크/해제 가능 -->
<input type="checkbox" id="agree" name="agree" />
<label for="agree">이용약관에 동의합니다</label>
```

<Paragraph>
  <strong>접근성 보너스</strong>: 스크린 리더가 input을 읽을 때 연결된 label의
  텍스트를 함께 읽어줍니다. label이 없으면 시각 장애인 사용자는 이 입력칸이 뭘
  위한 것인지 알 수 없습니다.
</Paragraph>

<Subtitle level={4}>`input` 태그의 `type` 속성</Subtitle>

<Paragraph>
  `type` 하나만 바꿔도 브라우저가 <strong>무료로 도와주는 기능</strong>이
  달라집니다. `type="text"`로 모든 걸 해결하려는 것은 만능 리모컨 대신 모든
  가전에 일일이 손을 대는 것과 같습니다.
</Paragraph>

| type       | 브라우저가 해주는 것    | 모바일 키보드   |
| :--------- | :---------------------- | :-------------- |
| `text`     | 없음                    | 일반 키보드     |
| `email`    | @ 포함 여부 자동 검증   | @ . 버튼 포함   |
| `password` | 입력값 마스킹(●)        | 일반 키보드     |
| `number`   | 숫자만 입력 가능        | 숫자 키패드     |
| `tel`      | 없음 (패턴은 직접 지정) | 전화번호 키패드 |

<Subtitle level={4}>`select`, `textarea`</Subtitle>

<Paragraph>
  `select`는 드롭다운 선택, `textarea`는 여러 줄 텍스트 입력을 담당합니다.
  `input`은 한 줄 입력에 특화되어 있으므로, 긴 텍스트(자기소개, 문의 내용
  등)에는 반드시 `textarea`를 사용합니다.
</Paragraph>

<Paragraph>
  `textarea`를 쓸 때 한 가지 주의할 점이 있습니다. 아래처럼 태그 사이에
  줄바꿈이나 들여쓰기를 넣으면, 그 공백이{" "}
  <strong>초기값으로 그대로 들어갑니다.</strong>
</Paragraph>

```html
<!-- 태그 사이의 개행/공백이 초기값으로 들어감 -->
<textarea name="content"> </textarea>

<!-- 이렇게 붙여써야 빈 상태로 시작함 -->
<textarea name="content"></textarea>
```

<Paragraph>
  이 문제는 `input`에서는 발생하지 않습니다. `input`은 자기 닫힘 태그이고 값은
  `value` 속성으로 지정하기 때문입니다. `textarea`는 태그 사이의 내용이 곧
  값이기 때문에 개행 문자가 끼어들 수 있습니다.
</Paragraph>

<Paragraph>
  나중에 JavaScript를 배우면 이런 상황에서 <strong>`trim()`</strong>이라는
  메서드를 자주 쓰게 됩니다. `trim()`은 문자열 앞뒤의 공백과 개행을 제거합니다.
</Paragraph>

```js
const value = document.querySelector("textarea").value;
// 사용자가 입력한 값: "\n안녕하세요\n"
console.log(value.trim()); // "안녕하세요"
```

<Paragraph>
  왜 제거해야 할까요? 앞뒤 공백이 붙은 채로 서버에 저장되면, 나중에 값을
  비교하거나 검색할 때 오작동이 생깁니다. 예를 들어 사용자가 아이디로 "seojin"을
  입력했는데 실제로 저장된 값이 " seojin"이라면, 로그인 비교 시 불일치가
  발생합니다. 사용자 입장에서는 올바르게 입력했는데 오류가 나는 것처럼 보이죠.
  `trim()`은 이런{" "}
  <strong>눈에 보이지 않는 공백 버그를 예방하는 가장 기본적인 습관</strong>
  입니다.
</Paragraph>

<Subtitle level={3}>CSS 선택자 핵심</Subtitle>

<Subtitle level={4}>기본 선택자 정리</Subtitle>

```css
/* 태그 선택자: 해당 태그 전부 선택 */
p {
  color: black;
}

/* 클래스 선택자: 마침표(.)로 시작, 여러 요소에 재사용 가능 */
.highlight {
  background: yellow;
}

/* 아이디 선택자: 샵(#)으로 시작, 페이지에서 딱 하나 */
#main-title {
  font-size: 2rem;
}
```

<Paragraph>
  <strong>헷갈리는 포인트</strong>: 클래스는 한 요소에 여러 개 붙일 수 있지만
  (`class="btn btn-primary"`), 아이디는 페이지에서 <strong>단 하나</strong>여야
  합니다. 아이디가 중복되면 HTML 문법 위반이고, JavaScript의 `getElementById`도
  첫 번째 요소만 반환합니다.
</Paragraph>

<Subtitle level={4}>자손 결합자 vs 자식 결합자</Subtitle>

```html
<nav>
  <ul>
    <li><a href="/">홈</a></li>
    <li>
      <div>
        <a href="/about">소개</a>
      </div>
    </li>
  </ul>
</nav>
```

```css
/* 자손 결합자: nav 안의 모든 a를 선택 (깊이 무관) */
nav a {
  color: white;
}

/* 자식 결합자: nav의 직계 자식 a만 선택 (여기서는 해당 없음) */
nav > a {
  color: white;
}
```

<Paragraph>
  위 HTML에서 `nav a`는 "홈"과 "소개" 링크 <strong>모두</strong> 선택합니다.
  하지만 `nav > a`는 `nav`의 직계 자식인 `a`가 없으므로 <strong>아무것도
  선택하지 않습니다</strong>. 자손 결합자(공백)는 "안에 있는 모든 것",
  자식 결합자(`>`)는 "바로 아래 한 단계"라고 기억하세요.
</Paragraph>

<Subtitle level={4}>CSS 우선순위 — 점수제로 이해하기</Subtitle>

<Paragraph>
  CSS 우선순위는 <strong>점수 계산</strong>으로 이해하면 직관적입니다. 같은
  요소에 여러 규칙이 적용될 때, 점수가 높은 쪽이 이깁니다.
</Paragraph>

| 종류                          | 점수    | 예시                              |
| :---------------------------- | :------ | :-------------------------------- |
| 인라인 스타일                 | 1,000점 | `style="color: red"`              |
| 아이디(#)                     | 100점   | `#header`                         |
| 클래스(.) / 속성 / 가상클래스 | 10점    | `.btn`, `[type="text"]`, `:hover` |
| 태그 / 가상요소               | 1점     | `div`, `p`, `::before`            |

```css
/* 점수: 태그(1) + 클래스(10) = 11점 */
p.intro {
  color: blue;
}

/* 점수: 아이디(100) = 100점 → 이쪽이 이김 */
#welcome {
  color: red;
}
```

<Paragraph>
  <strong>왜 id로 스타일링하면 안 좋을까?</strong> id 선택자는 100점이라
  클래스(10점)로는 절대 이길 수 없습니다. 나중에 스타일을 덮어써야 할 때 더 높은
  우선순위(`!important` 등)를 남발하게 되고, CSS가 점점 관리 불가능해집니다.
  그래서 실무에서는 <strong>스타일링에는 클래스만 사용</strong>하고, id는
  JavaScript 선택이나 앵커 링크 용도로만 쓰는 것이 관례입니다.
</Paragraph>

<Paragraph>
  <strong>같은 점수라면?</strong> 나중에 작성된(아래에 있는) 스타일이 이깁니다.
  이것을 <strong>캐스케이드(Cascade)</strong>라고 합니다. CSS의 C가 바로 이
  Cascade입니다.
</Paragraph>

<Subtitle level={4}>GET vs POST — 엽서 vs 편지 봉투</Subtitle>

<Paragraph>
  둘 다 데이터를 서버로 보내지만, <strong>데이터를 싣는 위치</strong>가
  다릅니다. GET은 <strong>엽서</strong>처럼 내용이 겉면(URL)에 다 보이고, POST는
  <strong>편지 봉투</strong>처럼 내용이 안에 들어 있습니다.
</Paragraph>

```
GET 요청:  https://example.com/search?q=프론트엔드&page=1
                                       ↑ 데이터가 URL에 노출

POST 요청: https://example.com/login
           Body: { username: "seojin", password: "1234" }
                  ↑ 데이터가 본문에 숨겨짐
```

<Paragraph>
  "주소창에 보여도 괜찮으면 GET, 아니면 POST"가 판단 기준입니다. GET은 URL 공유,
  북마크, 브라우저 캐싱이 가능하다는 고유한 장점이 있어서 검색과 필터에
  적합합니다.
</Paragraph>

<Subtitle level={4}>`fieldset`과 `legend`</Subtitle>

<Paragraph>
  회원가입처럼 입력 필드가 많을 때 "기본 정보", "보안 설정"처럼 관련 항목을
  그룹화합니다. 스크린 리더가 `legend`를 먼저 읽어주기 때문에 접근성에도 도움이
  됩니다.
</Paragraph>

```html
<fieldset>
  <legend>기본 정보</legend>
  <label for="name">이름</label>
  <input type="text" id="name" name="name" />
  <label for="email">이메일</label>
  <input type="email" id="email" name="email" />
</fieldset>
```

<Subtitle level={4}>CSS 적용 세 가지 방법</Subtitle>

<Paragraph>
  인라인 스타일(`style=""`), 내부 스타일시트(`<style>`), 외부 스타일
  시트(`<link>`) 중 <strong>외부 스타일 시트</strong>가 유지보수와
  재사용성 면에서 가장 권장됩니다. 인라인 스타일은 우선순위가 1,000점으로 너무
  높아 나중에 덮어쓰기가 극도로 어려워지므로 가급적 피합니다.
</Paragraph>

<Subtitle level={4}>벤더 프리픽스</Subtitle>

<Paragraph>
  `-webkit-`, `-moz-` 같은 접두사는 브라우저가 실험적 CSS 기능을 제공할 때
  붙입니다. 현재는 Autoprefixer가 빌드 시 자동으로 처리해주므로 직접 작성할 일은
  거의 없고, Can I Use 사이트로 브라우저 지원 현황을 확인할 수 있습니다.
</Paragraph>

<Subtitle level={4}>CSS 변수 (Custom Properties)</Subtitle>

<Paragraph>
  `--color-primary: #3b82f6`처럼 `--`로 시작하는 변수를 `:root`에 정의하고,
  `var(--color-primary)`로 참조합니다. 한 곳에서 값을 바꾸면 사용하는 모든 곳에
  반영되어 유지보수가 편합니다. 다크 모드 구현에도 활용됩니다.
</Paragraph>

```css
:root {
  --color-primary: #3b82f6;
  --color-text: #1f2937;
}

.btn {
  background: var(--color-primary);
  color: var(--color-text);
}
```

<Subtitle level={4}>의사 클래스 (`:hover`, `:focus`, `:nth-child`)</Subtitle>

<Paragraph>
  요소의 <strong>특정 상태</strong>를 선택하는 선택자입니다. 마우스를 올렸을
  때(`:hover`), 키보드 포커스가 갔을 때(`:focus`), n번째 자식(`:nth-child`) 등
  상태 기반 스타일링에 사용합니다. Tailwind에서는 `hover:`, `focus:` 접두사로
  적용합니다.
</Paragraph>

<Subtitle level={4}>폼 유효성 검증 속성</Subtitle>

<Paragraph>
  `required`(필수 입력), `pattern`(정규표현식), `minlength`/`maxlength`(글자 수
  제한) 등 <strong>HTML 속성만으로 기본적인 폼 검증이 가능</strong>합니다.
  JavaScript를 한 줄도 쓰지 않고도 "이 칸은 필수입니다"를 구현할 수 있습니다.
</Paragraph>

```html
<!-- 필수 입력 + 최소 8자 + 영문/숫자만 허용 -->
<input
  type="text"
  name="username"
  required
  minlength="8"
  pattern="[a-zA-Z0-9]+"
  title="영문과 숫자만 사용 가능합니다"
/>
```

<Subtitle level={2}>2. 저번 주 과제 정답 및 풀이</Subtitle>

<Subtitle level={3}>인증 미션 과제</Subtitle>

<Paragraph>
  <strong>
    Q1. 네이버나 구글에서 검색을 해보고, 주소창 URL을 확인해보세요. 어떤
    방식(GET/POST)을 사용하고 있나요? 그리고 왜 로그인 폼은 같은 방식을 쓰지
    않는다고 생각하나요?
  </strong>
</Paragraph>

<Subtitle level={4}>검색은 왜 GET인가?</Subtitle>

<Paragraph>
  검색창에 "리액트"를 입력하면 URL이{" "}
  <strong>`?q=리액트`처럼 쿼리스트링에 검색어가 붙습니다.</strong> 이것이 GET
  방식입니다.
</Paragraph>

- <strong>북마크·공유 가능</strong>: URL 자체가 검색 결과를 담고 있어서 링크를
  복사해 공유하면 상대방도 같은 결과를 볼 수 있습니다.
- <strong>캐싱 가능</strong>: 브라우저와 서버가 결과를 캐시해 두어 반복 검색이
  빠릅니다.
- <strong>멱등성</strong>: 몇 번을 요청해도 서버 상태가 변하지 않습니다.

<Subtitle level={4}>로그인은 왜 POST인가?</Subtitle>

- <strong>URL 노출 금지</strong>: GET을 쓰면 비밀번호가 주소창에 그대로
  나타납니다. 브라우저 히스토리, 서버 로그, 공유 화면에 모두 노출됩니다.
- <strong>서버 상태 변경</strong>: 로그인 성공 시 세션이 생성되는 등 서버 상태가
  바뀝니다. 상태를 변경하는 요청은 POST가 맞습니다.
- <strong>HTTPS + POST</strong>: POST 바디는 HTTPS 암호화 구간에서 보호되어
  안전하게 전달됩니다.

---

<Paragraph>
  <strong>
    Q2. 아래 CSS 코드에서 `.btn` 텍스트는 최종적으로 무슨 색이 될까요? 그리고 왜
    그 색이 적용되는지, 어떤 규칙 때문인지 설명해보세요.
  </strong>
</Paragraph>

```css
p {
  color: green;
}
.btn {
  color: red;
}
#submit {
  color: blue;
}
```

```html
<p class="btn" id="submit">제출</p>
```

<Subtitle level={4}>정답: blue</Subtitle>

<Paragraph>
  CSS <strong>명시도(Specificity)</strong> 규칙에 따라 가장 높은 점수를 가진
  선택자가 이깁니다.
</Paragraph>

- <strong>`p` (태그 선택자)</strong>: 명시도 0-0-1 → green
- <strong>`.btn` (클래스 선택자)</strong>: 명시도 0-1-0 → red
- <strong>`#submit` (ID 선택자)</strong>: 명시도 1-0-0 → blue

<Paragraph>
  ID 선택자가 가장 높은 명시도를 가지므로 <strong>최종 색은 blue</strong>입니다.
  코드에서 선언 순서와 관계없이 명시도가 높은 규칙이 항상 우선합니다.
</Paragraph>

<Subtitle level={3}>Q1. GET과 POST의 차이를 설명해주세요.</Subtitle>

<Paragraph>
  둘 다 HTTP 요청 방식이지만, <strong>데이터를 실어 보내는 위치</strong>가
  다릅니다.
</Paragraph>

- <strong>GET</strong>: 데이터가 URL의 쿼리 스트링에 붙어서 전송됩니다
  (`?key=value`). URL에 노출되므로 민감하지 않은 데이터(검색, 필터링)에
  적합합니다. URL 공유, 북마크, 브라우저 캐싱이 가능한 것이 장점입니다.
- <strong>POST</strong>: 데이터가 HTTP 요청 본문(Body)에 담겨 전송됩니다. URL에
  노출되지 않아 로그인, 회원가입, 결제 등 민감한 정보에 적합합니다.

<Paragraph>
  면접 팁: "주소창에 보여도 괜찮으면 GET, 아니면 POST"라고 정리하면 간결합니다.
  다만 POST도 HTTPS 없이는 안전하지 않다는 점도 언급하면 좋습니다.
</Paragraph>

<Subtitle level={3}>
  Q2. CSS 선택자 우선순위(Specificity)를 설명해주세요.
</Subtitle>

<Paragraph>
  같은 요소에 여러 CSS 규칙이 적용될 때, 어떤 스타일이 최종 적용될지 결정하는
  규칙입니다.
</Paragraph>

- <strong>우선순위 순서</strong>: `!important` > 인라인 스타일 > 아이디(#) >
클래스(.) / 속성 / 가상 클래스 > 태그 / 가상 요소
- 같은 우선순위라면 <strong>나중에 작성된 스타일</strong>이 적용됩니다.
- `!important`는 우선순위를 강제로 끌어올리지만, 디버깅이 극도로 어려워지므로
  실무에서는 가급적 사용하지 않습니다.

<Subtitle level={3}>Q3. 박스 모델(Box Model)이 무엇인가요?</Subtitle>

<Paragraph>
  모든 HTML 요소는 사각형 상자로 이루어져 있습니다. 이 상자는 안쪽부터
  바깥쪽으로 네 영역으로 구성됩니다.
</Paragraph>

1. <strong>Content</strong>: 텍스트, 이미지 등 실제 내용이 들어가는 영역
2. <strong>Padding</strong>: 콘텐츠와 테두리 사이의 안쪽 여백 (배경색 적용됨)
3. <strong>Border</strong>: 요소의 테두리
4. <strong>Margin</strong>: 테두리 바깥쪽의 다른 요소와의 간격 (투명)

<Paragraph>
  `box-sizing: content-box`(기본값)에서는 width가 콘텐츠 영역만을 의미하고,
  `box-sizing: border-box`에서는 padding과 border까지 포함합니다. 현대 웹
  개발에서는 `border-box`를 기본으로 설정하는 것이 표준입니다.
</Paragraph>

```css
/* content-box: width(200) + padding×2(40) + border×2(10) = 실제 250px */
.box {
  width: 200px;
  padding: 20px;
  border: 5px solid;
}

/* border-box: 실제 너비 200px 고정, 콘텐츠 = 200 - 40 - 10 = 150px */
.box {
  box-sizing: border-box;
  width: 200px;
  padding: 20px;
  border: 5px solid;
}
```
