<Subtitle level={2}>1. 저번 주 복습 — 컴포넌트, JSX, props</Subtitle>

<Paragraph>
  지난주에는 React를 처음 시작하며 컴포넌트, JSX, props를 배웠습니다. 핵심만
  빠르게 짚어보겠습니다.
</Paragraph>

<Subtitle level={3}>컴포넌트</Subtitle>

- <strong>함수형 컴포넌트</strong>: props를 받아 JSX를 반환하는 JS 함수
- <strong>이름은 대문자</strong>: 소문자이면 HTML 태그로 처리됨
- <strong>재사용 가능</strong>: 레고처럼 조합해서 복잡한 UI 구성

<Subtitle level={3}>JSX 핵심 규칙</Subtitle>

- <strong>{} 안에 JS 표현식</strong>: `{name}`, `{1 + 1}`, `
  {isAdmin && <Badge />}`
- <strong>class → className</strong>: `class`는 JS 예약어
- <strong>Fragment</strong>: `<>...</>`로 감싸면 불필요한 DOM 노드 없이 여러
  요소 반환

<Subtitle level={3}>props</Subtitle>

- <strong>부모 → 자식 단방향</strong>: 자식은 읽기만 가능, 수정 불가
- <strong>구조 분해로 수신</strong>: `function Card({ title, desc }) {}`
- <strong>key prop</strong>: 리스트 렌더링 시 필수 — 고유한 id 사용

<Subtitle level={2}>2. 저번 주 과제 정답 및 풀이</Subtitle>

<Subtitle level={3}>대면 11주차 인증 미션</Subtitle>

<Paragraph>
  <strong>Q1. JSX에서 JS 표현식 사용 방법과 조건부 렌더링 2가지</strong>
</Paragraph>

<Paragraph>
  JSX에서 JS 표현식은 `{}` 안에 작성합니다. 조건부 렌더링 방법: (1) `&&` 연산자:
  조건이 `true`일 때만 렌더링합니다. `{isLoggedIn && <Dashboard />}` (2) 삼항
  연산자: true/false에 따라 다른 것을 렌더링합니다. `
  {isLoggedIn ? <Dashboard /> : <Login />}`
</Paragraph>

<Paragraph>
  <strong>Q2. 컴포넌트 이름이 대문자여야 하는 이유</strong>
</Paragraph>

<Paragraph>
  소문자는 HTML 태그, 대문자는 컴포넌트로 구분합니다. JSX가
  `React.createElement()`로 변환될 때 소문자는 문자열(`'button'`)로, 대문자는
  함수 참조(`Button`)로 전달됩니다.
</Paragraph>

<Paragraph>
  <strong>Q3. props가 읽기 전용이어야 하는 이유</strong>
</Paragraph>

<Paragraph>
  React는 <strong>단방향 데이터 흐름(부모 → 자식)</strong>을 원칙으로 합니다.
  자식이 props를 수정하면 데이터 흐름이 깨지고, 어느 컴포넌트가 데이터를
  변경했는지 추적하기 어려워집니다. 데이터 변경은 반드시 소유자(부모)에서만 해야
  합니다.
</Paragraph>

<Subtitle level={2}>3. useState — 컴포넌트가 기억하는 값</Subtitle>

<Paragraph>
  일반 JS 변수는 컴포넌트가 리렌더링될 때마다 초기화됩니다.{" "}
  <strong>useState</strong>는 렌더링 사이에도 값을 유지합니다.
</Paragraph>

```jsx
import { useState } from "react";

function Counter() {
  // [현재 값, 값을 바꾸는 함수]
  const [count, setCount] = useState(0); // 초기값 0

  return (
    <div>
      <p>현재 카운트: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(count - 1)}>-1</button>
    </div>
  );
}
```

<Subtitle level={3}>state를 직접 변경하면 안 되는 이유</Subtitle>

```jsx
// ❌ 잘못된 방법 — React가 변경을 감지하지 못함
count++;
count = count + 1;

// ✅ 올바른 방법 — setState로만 변경
setCount(count + 1);
```

<Paragraph>
  React는 `setState`가 호출될 때 상태 변경을 감지하고 리렌더링합니다. 직접
  변경하면 React가 모르기 때문에 화면이 업데이트되지 않습니다.
</Paragraph>

<Subtitle level={3}>이전 상태 기반 업데이트 — 함수형 업데이트</Subtitle>

```jsx
// ❌ 클로저 함정 — 오래된 count를 참조할 수 있음
setCount(count + 1);

// ✅ 함수형 업데이트 — 항상 최신 상태를 받음
setCount((prev) => prev + 1);
```

<Paragraph>
  버튼을 빠르게 여러 번 클릭하거나 비동기 처리에서 상태를 업데이트할 때는{" "}
  <strong>함수형 업데이트</strong>를 사용해야 안전합니다.
</Paragraph>

<Subtitle level={3}>객체 상태 — 스프레드로 불변 업데이트</Subtitle>

```jsx
const [user, setUser] = useState({ name: "철수", age: 25 });

// ❌ 직접 수정 — 안 됨
user.age = 26;

// ✅ 스프레드로 새 객체 생성
setUser((prev) => ({ ...prev, age: prev.age + 1 }));
// { name: '철수', age: 26 }
```

<Paragraph>
  <strong>지난 주 클로저 + 불변성 개념이 여기서 만납니다.</strong> `prev =>`
  패턴은 클로저를 통해 최신 상태를 받고, 스프레드로 불변 업데이트를 합니다.
</Paragraph>

<Subtitle level={2}>4. useEffect — 렌더링 이후 실행되는 사이드 이펙트</Subtitle>

<Paragraph>
  <strong>사이드 이펙트</strong>란 컴포넌트 외부에 영향을 주는 작업입니다 — API
  호출, 타이머 설정, DOM 직접 조작 등. useEffect는 이런 작업을 렌더링 이후에
  실행합니다.
</Paragraph>

<Subtitle level={3}>기본 구조와 의존성 배열</Subtitle>

```jsx
import { useState, useEffect } from "react";

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // 렌더링 이후에 실행
    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then((data) => setUser(data));
  }, [userId]); // ← 의존성 배열: userId가 바뀔 때마다 재실행

  return <div>{user?.name}</div>;
}
```

<Subtitle level={3}>의존성 배열 3가지 패턴</Subtitle>

```jsx
// 1. 빈 배열 [] — 마운트 시 한 번만 실행
useEffect(() => {
  console.log("컴포넌트가 처음 나타났을 때 한 번만");
}, []);

// 2. 값이 있는 배열 [dep] — dep이 바뀔 때마다 실행
useEffect(() => {
  console.log("userId가 바뀔 때마다");
}, [userId]);

// 3. 배열 없음 — 매 렌더링마다 실행 (거의 안 씀)
useEffect(() => {
  console.log("렌더링할 때마다");
});
```

<Subtitle level={3}>async/await + try/catch 패턴</Subtitle>

```jsx
// useEffect 안에서 async 직접 사용 불가
// ❌ 잘못된 방법
useEffect(async () => {
  const data = await fetch("/api/data");
});

// ✅ 올바른 방법 — 내부에 async 함수 정의 후 호출
useEffect(() => {
  async function loadData() {
    try {
      const res = await fetch("/api/data");
      const data = await res.json();
      setData(data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }

  loadData();
}, []);
```

<Subtitle level={3}>클린업 함수 — 정리하기</Subtitle>

```jsx
useEffect(() => {
  const timer = setInterval(() => {
    setCount((prev) => prev + 1);
  }, 1000);

  // 클린업 함수 — 컴포넌트가 사라지거나 의존성이 바뀌기 전에 실행
  return () => clearInterval(timer);
}, []);
```

<Paragraph>
  타이머, 이벤트 리스너 등을 설정했다면 반드시 클린업 함수에서 제거해야 메모리
  누수를 방지할 수 있습니다.
</Paragraph>

<Subtitle level={3}>클로저와 stale closure 버그</Subtitle>

<Paragraph>
  useEffect와 클로저가 만나는 곳에서 버그가 자주 발생합니다. 의존성 배열에
  사용하는 값을 빠뜨리면 오래된 클로저(stale closure)를 참조합니다.
</Paragraph>

```jsx
// ❌ 버그 — count를 사용하지만 의존성에 없음
useEffect(() => {
  const timer = setInterval(() => {
    console.log(count); // 항상 초기값(0)을 출력
  }, 1000);
  return () => clearInterval(timer);
}, []); // count를 빠뜨림

// ✅ 해결 — 함수형 업데이트로 의존성 회피
useEffect(() => {
  const timer = setInterval(() => {
    setCount((prev) => prev + 1); // 함수형 업데이트로 count 의존성 제거
  }, 1000);
  return () => clearInterval(timer);
}, []);
```

<Subtitle level={2}>5. 면접 질문</Subtitle>

<Subtitle level={3}>
  Q1. useState에서 state를 직접 변경하면 안 되는 이유는?
</Subtitle>

<Paragraph>
  React는 setState 호출을 감지해서 리렌더링합니다. 직접 변경하면 React가 변경을
  모르기 때문에 화면이 업데이트되지 않습니다. 또한 불변성을 유지해야 이전
  상태와의 비교가 가능하고 React의 최적화 기능(memo, useMemo 등)이 올바르게
  동작합니다.
</Paragraph>

<Subtitle level={3}>Q2. useEffect의 의존성 배열은 왜 필요한가요?</Subtitle>

<Paragraph>
  의존성 배열은 effect가 <strong>언제 재실행될지</strong>를 제어합니다. 배열이
  없으면 매 렌더링마다, 빈 배열이면 마운트 시 한 번만, 값이 있으면 그 값이
  변경될 때만 실행됩니다. 의존성을 잘못 명시하면 stale closure 버그가 발생할 수
  있습니다.
</Paragraph>

<Subtitle level={3}>Q3. 빈 의존성 배열 []의 의미는?</Subtitle>

<Paragraph>
  컴포넌트가 처음 화면에 나타날 때(마운트) 한 번만 실행됩니다. API 초기 데이터
  로딩, 이벤트 리스너 등록 등에 주로 사용합니다. 컴포넌트가 화면에서 사라질
  때(언마운트) 클린업 함수가 실행됩니다.
</Paragraph>
