<Subtitle level={2}>1. 저번 주 복습</Subtitle>

<Paragraph>
  지난주에는 this 바인딩, 실행 컨텍스트, 이터러블과 Set/Map을 배웠습니다.
  핵심만 빠르게 짚어보겠습니다.
</Paragraph>

<Subtitle level={3}>this 바인딩</Subtitle>

- <strong>일반 함수 호출</strong>: 전역 객체 (`window`) — strict mode에서는 `undefined`
- <strong>메서드 호출</strong>: 호출한 객체
- <strong>생성자 호출</strong>: 새로 생성되는 인스턴스
- <strong>명시적 바인딩</strong>: `call` / `apply` (즉시 호출), `bind` (새 함수 반환)
- <strong>화살표 함수</strong>: 자체 `this` 없음 — 선언 시점 상위 스코프의 `this` 사용 (렉시컬 this)

<Subtitle level={3}>실행 컨텍스트</Subtitle>

- <strong>역할</strong>: 코드가 실행될 때 생성되는 환경 정보 묶음 — 변수, 스코프 체인, this 포함
- <strong>콜 스택</strong>: 함수 호출마다 실행 컨텍스트가 쌓이고, 종료 시 제거
- <strong>연결</strong>: 호이스팅·스코프·클로저 모두 실행 컨텍스트 메커니즘으로 설명 가능

<Subtitle level={3}>Set / Map</Subtitle>

- <strong>Set</strong>: 중복 없는 값의 집합. `[...new Set(arr)]`로 배열 중복 제거
- <strong>Map</strong>: 어떤 타입이든 키로 사용 가능. 삽입 순서 보장, `size` 프로퍼티 제공

<Subtitle level={2}>2. 저번 주 과제 정답 및 풀이</Subtitle>

<Subtitle level={3}>학습 자료 11주차 과제</Subtitle>

<Paragraph>
  <strong>Q1. JavaScript에서 this는 어떻게 결정되나요? 화살표 함수의 this가 일반 함수와 다른 이유를 포함해서 설명해주세요.</strong>
</Paragraph>

<Paragraph>
  `this`는 함수가 선언된 위치가 아니라 <strong>호출되는 방식</strong>에 따라 동적으로
  결정됩니다. 메서드 호출이면 호출한 객체, 일반 함수 호출이면 전역 객체, `new`이면 새
  인스턴스, `call/apply/bind`이면 명시적으로 지정한 객체입니다.
</Paragraph>

<Paragraph>
  화살표 함수는 자체 `this`를 가지지 않고 <strong>선언 시점의 상위 스코프 `this`를 캡처</strong>합니다.
  호출 방식에 영향받지 않으므로, 콜백이나 이벤트 핸들러에서 `this`를 유지해야 할 때
  유용합니다.
</Paragraph>

<Paragraph>
  <strong>Q2. 일반 객체({})와 Map의 차이를 설명하고, 어떤 상황에서 Map을 선택하는 게 나은지 예시와 함께 설명해주세요.</strong>
</Paragraph>

- <strong>키 타입</strong>: 일반 객체는 문자열/심볼만, Map은 어떤 타입이든 가능
- <strong>크기 확인</strong>: 일반 객체는 `Object.keys(obj).length`, Map은 `map.size`
- <strong>순서 보장</strong>: Map은 삽입 순서 보장, 일반 객체는 정수 키 먼저 정렬됨
- <strong>선택 기준</strong>: 키가 문자열이고 단순 데이터 저장이면 객체, 키 타입이 다양하거나 순서·크기가 중요하면 Map

<Subtitle level={3}>Q1. try/catch/finally는 각각 언제 실행되나요?</Subtitle>

- <strong>`try`</strong>: 에러가 발생할 수 있는 코드를 감싼 블록. 에러가 없으면 전부 실행
- <strong>`catch`</strong>: `try` 블록에서 에러가 발생했을 때 실행. 에러 객체를 인자로 받음
- <strong>`finally`</strong>: 에러 발생 여부와 관계없이 <strong>항상 실행</strong>. `return`이나 re-throw 후에도 실행

```js
async function loadData() {
  try {
    const data = await fetch("/api/data").then((r) => r.json());
    return data;
  } catch (err) {
    console.error("에러:", err.message);
    throw err; // 상위로 전파
  } finally {
    setLoading(false); // 성공/실패 관계없이 로딩 해제
  }
}
```

<Paragraph>
  면접 팁: "finally는 리소스 정리(파일 닫기, 로딩 스피너 해제 등)에 사용합니다. React에서
  데이터 페칭 시 `setLoading(false)`를 finally에 두면 성공·실패 양쪽에서 로딩 상태를
  안전하게 해제할 수 있습니다"라고 연결하면 좋습니다.
</Paragraph>

<Subtitle level={3}>Q2. 에러를 instanceof로 구별하는 이유는 무엇인가요?</Subtitle>

<Paragraph>
  `err.message`는 문자열이라 정확한 분기가 어렵습니다. 커스텀 에러 클래스를 만들어
  `Error`를 상속하면 `instanceof`로 에러 종류를 명확하게 구별할 수 있습니다.
</Paragraph>

```js
class NetworkError extends Error {
  constructor(msg, status) {
    super(msg);
    this.name = "NetworkError";
    this.status = status;
  }
}

try {
  throw new NetworkError("서버 오류", 500);
} catch (err) {
  if (err instanceof NetworkError) {
    // 네트워크 에러 전용 처리
  } else {
    throw err; // 모르는 에러는 다시 전파
  }
}
```

<Subtitle level={3}>Q3. 정규 표현식은 어떤 상황에서 사용하나요?</Subtitle>

<Paragraph>
  정규 표현식은 <strong>문자열의 패턴을 검사하거나 추출·치환</strong>할 때 사용합니다.
  실무에서 자주 쓰는 사례는 세 가지입니다.
</Paragraph>

- <strong>입력 유효성 검사</strong>: 이메일, 전화번호, 비밀번호 형식 검사
- <strong>문자열 파싱</strong>: URL에서 도메인 추출, 날짜 포맷 파싱
- <strong>포맷 변환</strong>: `YYYY-MM-DD` → `YYYY.MM.DD` 변환

<Paragraph>
  면접 팁: "정규 표현식을 무조건 쓰는 것보다, 단순한 경우는 `includes`, `startsWith`,
  `split` 등으로 처리하고 복잡한 패턴에만 정규 표현식을 사용하는 게 가독성이 좋습니다"
  라고 답하면 실무 감각이 있다는 인상을 줍니다.
</Paragraph>
