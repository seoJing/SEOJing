<Subtitle level={2}>0. 이 글의 역할</Subtitle>

<Paragraph>
  okayJing 글이 처음 몇 개일 때는 루트에 쌓아도 괜찮았다. 그런데 OpenClaw 시절
  기록, Hermes 전환, memory, workflow, voice, automation까지 같이 늘어나면서
  목록만 보고는 어디서부터 읽어야 하는지 알기 어려워졌다. 그래서 기준을 하나
  바꿨다.
</Paragraph>

<Paragraph>
  지금은 루트에 두 종류의 입구를 둔다. 기술 축과 목적별 읽기 순서를 먼저 잡고
  싶다면 [`오케이징 기술 지도`](/blog/okayJing/technical-map)를 읽는다. 이 글은
  폴더별 접근과 시간순 흐름을 확인하는 아카이브 가이드로 남긴다.
</Paragraph>

---

<Subtitle level={2}>1. 폴더별로 접근하는 법</Subtitle>

<Paragraph>
  폴더는 "기능 이름"보다 "어떤 질문을 해결하는가"를 기준으로 나눴다. 특정 글을
  찾고 싶을 때는 아래 질문에서 출발하면 된다.
</Paragraph>

| 폴더            | 먼저 보면 좋은 질문                                      | 내용                                                                           |
| --------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `evolution/`    | 오케이징 구조가 왜 이렇게 바뀌었나?                      | OpenClaw 시절, Hermes 단일 체계 전환, vNext 판단, migration 잔재 정리          |
| `workflow/`     | 실제 작업은 어떻게 티켓·세션·보고로 굴러가나?            | Discord Forum, hermes-ticket, 작업 트리거, MSW/DTO handoff, 승인 피로 정리     |
| `memory/`       | 오케이징은 무엇을 기억하고 무엇을 버리나?                | memory, ticket, session_search, source-linked context pack, local-first memory |
| `automation/`   | 밤사이와 아침에는 무엇이 자동으로 도는가?                | dreaming loop, self-contained cron prompt, 아침브리핑 운영                     |
| `skills/`       | 우로보로스나 낫징 같은 이름은 지금 어디에 남았나?        | 인격이 아니라 재사용 가능한 skill discipline으로 남은 절차들                   |
| `discord/`      | 사용자가 만나는 대화 표면은 어떻게 바뀌었나?             | Discord 이사, free-response, channel prompt, gateway 대화 정책                 |
| `voice/`        | 텍스트 봇이 아니라 말하는 동료가 되려면 무엇이 달라지나? | TTS 선택, 음성 문체, Supertonic3 튜닝, 음성 요청을 포스트로 회수하는 흐름      |
| `architecture/` | Jing Factory나 Jing Studio 같은 큰 구조는 어디로 가나?   | Hermes 시대의 제품/작업 공장 구조와 계약 중심 설계                             |

---

<Subtitle level={2}>2. 처음 읽는다면 이 순서가 편하다</Subtitle>

<Paragraph>
  처음부터 모든 폴더를 훑을 필요는 없다. 전체 맥락을 잡고 싶다면 아래 순서가
  가장 덜 튄다. OpenClaw에서 시작해 Hermes 단일 체계로 넘어오고, 그 다음에
  workflow와 memory가 붙는 흐름이다.
</Paragraph>

<Paragraph>
  아래 목록의 글 경로는 바로 이동할 수 있는 링크로 걸어두었다. 새 탭에서 열고
  싶으면 링크를 직접 열면 되고, 전체 순서를 따라가고 싶으면 3장의 시간순 표를
  따라가면 된다.
</Paragraph>

1. [`evolution/openclaw-operating-series-overview`](/blog/okayJing/evolution/openclaw-operating-series-overview) — 처음 okayJing 카테고리를 왜 만들었는지
2. [`evolution/openclaw-big-picture`](/blog/okayJing/evolution/openclaw-big-picture) — OpenClaw를 어떤 운영 추상화로 봤는지
3. [`evolution/openclaw-runtime-core`](/blog/okayJing/evolution/openclaw-runtime-core) — gateway, sessions, tools를 어떻게 이해했는지
4. [`evolution/openclaw-ops-decisions`](/blog/okayJing/evolution/openclaw-ops-decisions) — 초기에 어떤 운영 결정을 남겼는지
5. [`discord/discord-migration-and-hermes-identity`](/blog/okayJing/discord/discord-migration-and-hermes-identity) — Discord와 Hermes 정체성이 어떻게 잡혔는지
6. [`workflow/multi-agent-ticket-structure`](/blog/okayJing/workflow/multi-agent-ticket-structure) — 다중 에이전트 조율에서 왜 티켓이 필요했는지
7. [`workflow/forum-tickets-hermes-gateway`](/blog/okayJing/workflow/forum-tickets-hermes-gateway) — 채팅이 작업 티켓으로 바뀐 지점
8. [`evolution/openclaw-to-hermes-single-system`](/blog/okayJing/evolution/openclaw-to-hermes-single-system) — 결국 왜 Hermes 단일 체계로 접었는지
9. [`skills/ouroboros-planning-skill`](/blog/okayJing/skills/ouroboros-planning-skill) + [`skills/notjing-final-gate`](/blog/okayJing/skills/notjing-final-gate) — 이름은 skill로 내려온 과정
10. [`memory/memory-ticket-session-search`](/blog/okayJing/memory/memory-ticket-session-search) — memory, ticket, session_search의 역할 분리

<Paragraph>
  여기까지 읽으면 "오케이징이 왜 여러 인격 팀이 아니라 Hermes 안의 절차 묶음이
  됐는지"가 대략 보인다. 그 다음부터는 관심사별 폴더로 들어가면 된다.
</Paragraph>

---

<Subtitle level={2}>3. 시간순으로 읽고 싶다면</Subtitle>

<Paragraph>
  시간순으로 읽을 때는 폴더 구조를 잠깐 무시하고 날짜를 따라가면 된다. 다만
  날짜순은 당시의 시행착오를 그대로 따라가는 방식이라, 중간에 이름이나 구조가
  지금과 다르게 보일 수 있다. 그게 오히려 이 시리즈의 목적이기도 하다.
</Paragraph>

<Subtitle level={2}>4. 지금 기준으로 다시 읽을 때 주의할 점</Subtitle>

<Paragraph>
  초반 글에는 OpenClaw라는 이름이 많이 나온다. 지금의 active 구조는 Hermes 단일
  체계에 가깝다. 그래서 초반 글을 현재 운영 문서처럼 그대로 읽으면 조금 헷갈릴
  수 있다. 초반 글은 "지금 구조의 원형과 실패 기록"으로 읽는 편이 맞다.
</Paragraph>

<Paragraph>
  반대로 최근 글일수록 실제 운영 기준에 가깝다. memory, automation, voice,
  workflow 쪽 글은 현재 Mac mini Hermes 운영과 바로 연결된다. 그래서 빠르게 현재
  기준만 보고 싶다면 `workflow/`, `memory/`, `automation/`, `voice/`부터 읽어도
  된다.
</Paragraph>

<Paragraph>
  결국 이 카테고리는 완성된 매뉴얼이 아니라, 오케이징이 계속 자기 운영 방식을
  고쳐가며 남긴 변화 기록이다. 폴더는 그 기록을 덜 흩어지게 만들기 위한 장치다.
</Paragraph>
