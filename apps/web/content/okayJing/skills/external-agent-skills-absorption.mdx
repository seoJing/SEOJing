<Subtitle level={2}>0. 좋은 skill repo라도 그대로 맞지는 않는다</Subtitle>

<Paragraph>
  외부 agent skill repository를 보면 가져오고 싶은 패턴이 많다. description을
  trigger처럼 쓰는 방식, 큰 skill을 reference와 script로 나누는 방식, 작업
  생애주기를 명시하는 방식은 분명 유용하다.
</Paragraph>

<Paragraph>
  하지만 그대로 복사하면 안 된다. 그 repo의 전제, 말투, 권한 모델, 자동화 범위가
  OkayJing과 같지 않기 때문이다. 잘 만든 skill도 남의 운영 규칙이다.
</Paragraph>

---

<Subtitle level={2}>1. 흡수는 install이 아니라 번역이다</Subtitle>

```text
external repo pattern
→ useful idea 추출
→ OkayJing 운영 문맥으로 재작성
→ trigger / procedure / pitfall / verification으로 skill화
→ provenance와 security disposition 남김
```

<Paragraph>
  지금 OkayJing이 가져간 것은 원문이 아니라 기준이다. `description`은 장식이
  아니라 trigger contract다. 큰 skill은 progressive disclosure로 나눈다.
  Understand, Plan, Execute, Verify, Review, Report, Absorb 같은 lifecycle
  vocabulary는 작업을 닫는 순서를 분명히 하는 데 쓴다.
</Paragraph>

<Paragraph>
  다만 이 표현들도 그대로 권위가 되는 것은 아니다. OkayJing의 ticket, session,
  memory, cron, hub-spoke 구조 안에서 다시 써야 한다.
</Paragraph>

---

<Subtitle level={2}>2. 흡수 후에도 평가가 필요하다</Subtitle>

<Paragraph>
  좋은 문장을 skill에 넣었다고 끝이 아니다. 반복 작업에서 실제로 도움이 되는지,
  오류를 줄이는지, 보고 품질을 높이는지 봐야 한다. workflow trace나 작은 eval
  set이 필요한 이유도 여기에 있다.
</Paragraph>

<Paragraph>
  외부 skill을 빨리 늘리는 것보다, OkayJing 안에서 쓸 수 있는 규칙으로 번역하고
  검증하는 편이 낫다. 느려 보이지만 나중에 덜 위험하다.
</Paragraph>

---

<Subtitle level={2}>3. 복사보다 번역이 필요한 이유</Subtitle>

<Paragraph>
  다른 agent framework의 skill을 볼 때 중요한 것은 문장 그대로가 아니다. 그
  skill이 어떤 실패를 막으려는지, 어떤 입력을 요구하는지, 어떤 검증을 끝으로
  삼는지를 뽑아야 한다. 그런 뒤 자기 시스템의 도구 이름, 권한 모델, 보고 방식에
  맞게 다시 써야 한다.
</Paragraph>

<Paragraph>
  예를 들어 어떤 skill이 "테스트를 실행하라"고 말하더라도, 내 시스템의 실제
  계약은 더 구체적이어야 한다. 어떤 테스트가 CI와 같은지, 실패하면 어디까지 자동
  수정해도 되는지, 외부로 push하기 전에 어떤 근거를 남길지까지 정해야 한다.
  흡수는 수집이 아니라 운영 규칙으로 바꾸는 일이다.
</Paragraph>
